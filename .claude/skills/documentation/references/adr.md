# ADR (Architecture Decision Record) の運用

ADR は、将来の実装を縛る設計判断を、その文脈と結果とともに残す恒常系の記録である。「A を選んだ」という結論だけでなく、「B と C も検討したうえで A にした理由」まで残す。こうしておくと、後から決定を見直すときに同じ議論を繰り返さずに済み、前提が変わったときにも「どの前提が崩れたのか」から議論を再開できる。

## いつ書くか

将来にわたって実装や運用を縛る判断をしたときに書く。具体的には、複数の選択肢から 1 つを選んだとき、採用した技術・構成・方針が後続の変更の前提になるとき、後から「なぜこうなっているのか」と問われる可能性が高いときのいずれかに当てはまれば書く。目安として、「3 か月後に加わった新しいメンバーが『なぜこうなっているのか』と聞きそうか」を自問すると判断しやすい。

逆に、1 つの変更の中だけで完結する理由は、ADR にせず PR 本文に書く。PR に書いた判断理由のうち、将来を縛るものだけを ADR へ昇格させる。昇格の基準は後述する。

## 置き場と命名

`docs/adr/` に、1 つの決定につき 1 ファイルを置く。ファイル名は `YYYYMMDD-<ハイフン区切りの短い題>.md` とし、`YYYYMMDD` は作成日にする。1 決定 1 ファイルに分けるのは、決定を覆すときに置き換えの単位を明確にするためである。連番ではなく日付を使うのは、複数の ADR を並行して作成したときに番号の割り当てが git 上で衝突するのを避けるためである。最初の ADR としては、「ADR を使うことを決めた」という決定自体を `20260101-record-architecture-decisions.md` のように残すとよい。

## 形式

Nygard 形式に従い、次の節で構成する。本文テンプレートは [templates/adr.md](./templates/adr.md) にある。

- タイトル: `ADR-YYYYMMDD: <タイトル>` の形式で書く。
- 決定の状態: 後述の Status の値を書く。
- 本決定が必要になった背景と状況: 決定そのものではなく、決定を迫っている課題・制約・前提を書く。
- 検討した選択肢とそれぞれの長所・短所: 2 つ以上挙げ、「何もしない (現状維持)」を必ず含める。現状維持と比較して初めて、変更するだけの価値があるという説得力が生まれるためである。
- 採用する方針: 「〜を採用する」の形で明確に書く。
- その方針を採用し他を退けた理由: 採用案の決め手と、他の案を退けた具体的な理由を書く。
- 本決定によって生じる影響と後続の作業: 良い面と悪い面の両方を、受け入れるトレードオフも含めて書き、後続タスクがあれば列挙する。
- 関連する Issue・PR・ドキュメント (任意): 関連する Issue・PR・Design Doc・外部資料へリンクする。

ADR は短く保つ。1 つの決定に集中させ、1 ページ以内に収める。検討の全体が 1 ページに収まらない大きな設計は、Design Doc ([design_doc.md](./design_doc.md)) に切り出し、そこから将来を縛る決定だけを ADR にする。

## Status の値と遷移

Status は Proposed から Accepted へ進むのが基本の流れであり、役目を終えた決定は Deprecated か Superseded のどちらかになる。それぞれの値の意味は次のとおりである。

- Proposed: 提案中であり、レビューや議論の途中である。
- Accepted: 受理済みであり、現在有効な決定である。
- Deprecated: 非推奨になった。後継の決定はないが、現在は推奨しない。
- Superseded by ADR-YYYYMMDD: 別の ADR に置き換えられた。`Superseded by ADR-20260101 (2026-01-01)` のように、後継の ADR と置き換えた日付を記す。

## 受理後は編集せず supersede する

ADR は PR として提出し、レビューで Approve を得たら、マージする前の最終 push で Status を Accepted に変更する (development-flow スキルの merge_and_cleanup.md を参照)。Accepted になった ADR の本文は編集しない。ただし、誤字の修正やリンク切れの補修は許容する。決定を覆すときは新しい ADR を書き、旧 ADR の Status を `Superseded by ADR-YYYYMMDD` に変えて、双方を相互にリンクする。こうすることで決定の正本が常に 1 つに保たれ、「いつからいつまで、どの決定が有効だったか」という履歴も失われずに残る。

## PR の判断理由からの昇格基準

PR 本文に書いた実装方針の判断理由は、その変更限りの説明であれば PR に留めてよい。ただし、その判断が今後の別の変更でも前提として参照される場合、その判断を覆すと広い範囲に影響が及ぶ場合、後から「なぜこの構成なのか」と繰り返し問われる見込みがある場合は、ADR へ昇格させ、PR からはその ADR へリンクする。
