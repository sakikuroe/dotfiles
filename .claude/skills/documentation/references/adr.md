# ADR (Architecture Decision Record) の運用

ADR は、将来の実装を縛る設計判断を、その文脈と結果とともに残す恒常系の記録である。「A を選んだ」という結論だけでなく、「B と C を検討して A にした理由」を残すことで、後から決定を見直すときに同じ議論を繰り返さずに済む。

## いつ書くか

次のような、将来にわたって実装や運用を縛る判断をしたときに書く。

- 複数の選択肢があり、その中から 1 つを選んだとき。
- 採用した技術・構成・方針が、後続の変更の前提になるとき。
- 後から「なぜこうなっているのか」と問われる可能性が高いとき。

1 つの変更の中だけで完結する理由は、ADR ではなく PR 本文に書く。PR の「実装方針の判断理由」に書いた内容のうち、将来の実装を縛るものだけを ADR へ昇格させる。

## 置き場と命名

- `docs/adr/` に、1 つの決定につき 1 ファイルを置く。
- ファイル名は `NNNN-<ハイフン区切りの短い題>.md` とする。`NNNN` は `0001` から始まる連番。
- 最初の ADR として、「ADR を使うことを決めた」という決定自体を `0001-record-architecture-decisions.md` に残すとよい。

## 形式

Nygard 形式に従い、次の節で構成する。本文テンプレートは [templates/adr.md](./templates/adr.md) にある。

- タイトル: `ADR-NNNN: <タイトル>`
- 決定の状態: 後述の値
- 本決定が必要になった背景と状況: 決定を迫っている背景・課題・制約を述べる。決定そのものではなく、決定を必要とする状況を記載する。
- 検討した選択肢とそれぞれの長所・短所: 最低 2 つ以上挙げる。「何もしない（現状維持）」を必ず含める。
- 採用する方針: 「〜を採用する」の形で明確に書く。
- その方針を採用し他を退けた理由: 採用した方針の決め手と、他を退けた具体的な理由を記す。
- 本決定によって生じる影響と後続の作業: 良い面・悪い面の両方と、受け入れるトレードオフを記す。後続タスクがあれば列挙する。
- 関連する Issue・PR・ドキュメント（任意）: 関連する Issue・PR・Design Doc・外部資料へのリンク

ADR は短く保つ。1 つの決定に集中し、1 ページ以内に収める。収まらない場合は Design Doc（docs/design/）として独立した文書にし、その判断を ADR として切り出す。

## Status の値と遷移

- Proposed: 提案中であり、レビューや議論の途中である。
- Accepted: 受理済みであり、現在有効な決定である。
- Deprecated: 非推奨になった。後継はないが、もう推奨しない。
- Superseded by ADR-XXXX: 別の ADR に置き換えられた。後継 ADR を `ADR-NNNN` 形式で記す (例: `Superseded by ADR-0007（2026-01-01）`)。

## 受理後は編集せず supersede する

ADR は PR として提出し、レビュー・Approve を経て Accepted にする。Accepted になった ADR の本文は編集しない。ただし、誤字の修正やリンク切れの補修は許容する。決定を覆す場合は、新しい ADR を書き、旧 ADR の Status を `Superseded by ADR-XXXX` に変え、双方を相互リンクする。こうすることで、決定の正本が常に 1 つに保たれ、「いつからいつまで、どの決定が有効だったか」の履歴も残る。

## PR の判断理由からの昇格基準

PR 本文の「実装方針の判断理由」に書いた内容は、その変更限りの説明であれば PR に留める。次のいずれかに当てはまる場合は、ADR へ昇格させ、PR からはその ADR へリンクする。

- その判断が、今後の別の変更でも前提として参照されること。
- その判断を覆すと、広い範囲に影響が及ぶこと。
- 後から「なぜこの構成なのか」と繰り返し問われる見込みがあること。
