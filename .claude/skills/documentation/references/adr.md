# ADR (Architecture Decision Record) の運用

ADR は、将来の実装を縛る設計上の判断を、その背景にある文脈や結果と併せて残す記録である。「A を選んだ」という結論だけでなく、「B や C を検討したうえで A を選んだ理由」まで記述する。こうすることで、後から決定を見直す際に同じ議論を繰り返さずに済み、前提となる状況が変わったときにも「当時のどの前提が崩れたのか」を起点として議論を再開できる。

## いつ書くか

将来にわたって実装や運用を縛る判断を下したときに書く。具体的には、「複数の選択肢から 1 つを選んだとき」「採用した技術・構成・方針が今後の変更の前提になるとき」「後から『なぜこうなっているのか』と問われる可能性が高いとき」のいずれかに当てはまる場合に作成する。「3 か月後に新しく加わったメンバーが『なぜこうなっているのか』と尋ねてきそうか」を自問してみると判断しやすい。

反対に、1 つの変更範囲内のみで完結する理由は ADR には書かず、PR の本文に記載する。PR に書いた判断理由のうち、将来の実装を縛るものだけを ADR へ昇格させる。昇格の基準については後述する。

## 置き場と命名

`docs/adr/` ディレクトリに、1 つの決定につき 1 つのファイルを置く。ファイル名は `YYYYMMDD-<ハイフン区切りの短いタイトル>.md` とし、`YYYYMMDD` には作成日を入れる。1 決定 1 ファイルに分割するのは、決定を覆す際の置き換え単位を明確にするためである。また、連番ではなく日付を使用するのは、複数の ADR を並行して作成した際に Git 上で番号の割り当てが衝突するのを防ぐためである。最初の ADR として、「ADR の運用を開始する」という決定自体を `20260101-record-architecture-decisions.md` のように残しておくとよい。

## 形式

Nygard 形式に従い、以下の項目で構成する。本文のテンプレートは [templates/adr.md](./templates/adr.md) に用意してある。

- タイトル: `ADR-YYYYMMDD: <タイトル>` の形式で記述する。
- 決定の状態: 後述する Status の値を記述する。
- 本決定が必要になった背景と状況: 決定内容そのものではなく、意思決定を迫られている課題や制約、前提条件を記述する。
- 検討した選択肢とそれぞれの長所・短所: 2 つ以上の選択肢を挙げ、「何もしない (現状維持)」を必ず含める。現状と比較して初めて、変更を加えるだけの価値があるという説得力が生まれるためである。
- 採用する方針: 「〜を採用する」という形で明確に記述する。
- その方針を採用し他を退けた理由: 採用案の決め手と、他の案を見送った具体的な理由を記述する。
- 本決定によって生じる影響と後続の作業: 良い面と悪い面の両方を、受け入れるトレードオフも含めて記述し、後続タスクがあれば列挙する。
- 関連する Issue・PR・ドキュメント (任意): 関連する Issue や PR、Design Doc、外部資料などへのリンクを記載する。

ADR は簡潔に保つ。1 つの決定事項に的を絞り、1 ページ以内に収める。検討事項全体が 1 ページに収まらない大規模な設計は、Design Doc ([design_doc.md](./design_doc.md)) に切り出し、その中から将来を縛る決定だけを抽出して ADR にする。

## Status の値と遷移

Status は Proposed から Accepted へ進むのが基本的な流れであり、役目を終えた決定は Deprecated または Superseded になる。それぞれの値の意味は以下のとおりである。

- Proposed: 提案中であり、レビューや議論の途中であることを示す。
- Accepted: 受理済みであり、現在有効な決定であることを示す。
- Deprecated: 非推奨になったことを示す。後継の決定はないが、現在は推奨されない状態である。
- Superseded by ADR-YYYYMMDD: 別の ADR に置き換えられたことを示す。`Superseded by ADR-20260101 (2026-01-01)` のように、後継の ADR と置き換えた日付を併記する。

## 受理後は編集せず supersede する

ADR は PR として提出し、レビューで Approve を得たら、マージする前の最後の push で Status を Accepted に変更する (development-flow スキルの merge_and_cleanup.md を参照)。Accepted になった ADR の本文は編集してはいけない。ただし、誤字脱字の修正やリンク切れの解消といった軽微な修正は許容される。過去の決定を覆す場合は、新しく ADR を書いたうえで古い ADR の Status を `Superseded by ADR-YYYYMMDD` に変更し、双方のファイルから相互にリンクを張る。これにより、有効な決定の正本が常に 1 つに保たれるとともに、「いつからいつまで、どの決定が有効だったか」という履歴を確実に残すことができる。

## PR の判断理由からの昇格基準

PR の本文に記述した実装方針の判断理由は、その変更のみに関する説明であれば PR に留めておいてよい。しかし、その判断が今後の別の変更で前提として参照される場合や、判断を覆した際に広範囲に影響が及ぶ場合、あるいは後から「なぜこの構成になっているのか」と繰り返し問われる可能性が高い場合は、ADR へ昇格させ、PR からは該当する ADR へリンクを張るようにする。
