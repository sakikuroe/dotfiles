# 置き場マップ

機能コード以外のすべての情報について、「どこに書き、どこに書かないか」を定める。系ごとの対照表が「置き場マップ」の正本であり、表の下で各行の判断根拠と実例を詳述する。情報をどこに書くか迷った際の、最終的な判断基準とする。

判断の出発点は常に同じである。その情報が「1 つの変更に紐づく記録 (変更系)」なのか、「現在の状態を表す記録 (恒常系)」なのかをまず見極め、該当する系の表から置き場を選ぶ。変更系は完了後に凍結し、恒常系は変更のたびに更新する。

各行の実例では、「アップロード API にレート制限を導入する」という 1 つの変更を一貫して用いる。同じ変更から生まれた情報が、その性質に応じて別々の置き場へと散っていく様子を追えるようにするためである。

## 変更系 (1 変更ごとに作り、完了後は凍結する)

| 置き場所 | ここに書く (正本) | ここには書かない (→ 正しい置き場) |
| --- | --- | --- |
| Issue 本文 | 課題・背景・動機、利用者視点の期待動作、受け入れ基準 (完了条件)、スコープ外 | 実装手段・API・設定・アーキテクチャの全体像 (→ docs)、進捗 (→ Issue コメント) |
| Issue コメント (実装計画) | 実装方針、変更対象、検討した代替案と却下理由、テスト方針 | 確定後の現在仕様 (→ reference)、将来を縛る決定 (→ ADR) |
| Issue コメント (進捗) | ブランチ名・PR・状態 | 設計・仕様・知見 (→ 各正本) |
| PR 本文 | 対応 Issue、変更概要、計画からの乖離点 (計画へリンク)、スコープ外、動作確認・テスト結果、恒常系 docs 更新の確認 | Issue 背景の再掲 (→ Issue へリンク)、現在仕様の全体 (→ docs)、決定の正本 (→ ADR、PR はリンク) |
| コミットメッセージ | 1 関心事の変更内容とその理由 | 大きな設計判断の正本 (→ ADR) |
| Design Doc (`docs/design/`) | 1 ページの ADR に収まらない大きな変更の背景、目標と非目標、設計案、代替案の比較、移行計画 | 将来を縛る決定の正本 (→ ADR へ切り出す)、確定後の現在仕様 (→ reference) |

### Issue 本文には、なぜ必要で何をどうしたいかを書く

課題・背景・動機、利用者視点での期待する動作、受け入れ基準 (完了条件)、スコープ外の事項を書く。Issue 本文は、「なぜ必要か」と「何をどうしたいか」を利用者の視点で述べる場所であり、第三者がこれを読むだけで対応の要否と優先度を判断できる状態を目指す。一度書いたら、以後は原則として書き換えない。書き換えてしまうと、後から読む人が「当時、何を課題としていたか」を復元できなくなるためである。

> 例: 「短時間の連続アップロードでサーバーが過負荷になる。1 ユーザーあたり毎分 60 リクエストまでに制限したい。受け入れ基準: 61 回目のリクエストが 429 で拒否される。スコープ外: ダウンロード API の制限」

実装手段・API 仕様・設定値・アーキテクチャの全体像は書かない。これらはコードや docs から常に復元できる情報である。Issue にも書いてしまうと二重管理になり、変更のたびにどちらかが陳腐化する (→ docs)。上の例でいえば、「Redis のトークンバケットで実装する」というのは実装手段であるため、Issue 本文ではなく実装計画のコメントに書く。進捗も本文には書かず、コメントに時系列で残す (→ Issue コメント)。

### Issue コメント (実装計画) で、着手前にどう作るかを合意する

着手前の実装方針、変更対象のファイルやモジュール、検討した代替案とその却下理由、テスト方針を書く。コードを大量に書いた後で方針の誤りに気づくといった手戻りを防ぐための、着手前の合意形成が目的である。計画はあくまで予測であるため、実装結果と食い違ってもこのコメントは書き換えず、実際の変更との乖離は PR 本文に書く (→ PR 本文)。

> 例: 「Redis のトークンバケットで実装し、ミドルウェア層に置く。代替案として Nginx の rate limit も検討したが、ユーザー単位の制御ができないため却下する。テストは時刻を固定した単体テストで補充ロジックを検証する」

確定した後の「現在の仕様」は書かない (→ reference)。また、計画に書いた判断のうち、今回の変更以外にも影響が及ぶものは ADR へ昇格させる (→ ADR)。上の例でいえば、「レート制限の状態を Redis に置く」という判断は、今後の別の機能開発でも前提となるため ADR に切り出す。

### Issue コメント (進捗) には、いまどの段階にいるかだけを書く

ブランチ名・PR 番号・状態の変化だけを書く。作業の中断や再開時に、コメントの履歴を上から読むだけで「いまどの段階にいるか」を把握できるようにするためである。

> 例: 「ブランチ feature/rate-limit で着手」「PR #123 をオープン」

設計・仕様・知見は書かない。進捗コメントは流れていく記録であり、後から検索して見つけ出すことを想定していないため、ここに書いた知見は事実上失われてしまう。作業中に得た知見はそれぞれの正本 (コードコメント・docs・ADR) に書き、ここには状態の遷移だけを残す。

### PR 本文には、この変更で実際に何をしたかを書く

対応する Issue、変更の概要、実装計画からの乖離点、スコープ外の事項、動作確認・テストの結果、影響を受ける恒常系ドキュメントを更新したかの確認を書く。乖離点については計画コメントへのリンクを添え、変わった点だけを書く。計画どおりの部分まで再掲すると、レビュアーが本当に確認すべき差分を見つけにくくなるためである。

> 例: 「Issue #100 に対応。計画ではミドルウェア層としていたが、ユーザーの特定に認証情報が必要なためハンドラー直前に変更した (計画コメントへリンク)。61 回目のリクエストが 429 になることを手元で確認済み。docs/reference/api.md に上限値を追記した」

Issue の背景は再掲せず、Issue へのリンクを張る。現在の仕様の全体像も書かない (→ docs)。将来を縛る決定事項の正本もここには置かず、ADR に記載したうえで PR からリンクする (→ ADR)。PR は一度マージされると凍結される変更記録であるため、ここを正本にしてしまうと、後から状態が変わった際に追従できなくなる。

### コミットメッセージには、1 つの関心事の変更内容とその理由を書く

1 つの関心事に対する変更内容と、その変更を行った理由を書く。関心事を 1 つに絞るのは、切り戻しやレビューをコミット単位で行えるようにするためである。

> 例: 「rate limiter の上限を設定ファイルから読むようにする。環境ごとに閾値を変えられるようにするため」

大きな設計判断の正本は置かない (→ ADR)。コミットメッセージは個々の変更に紐づく記録であり、横断的な決定事項をここに書くと、後からその決定に辿り着くために `git log` を遡らなければならなくなる。

### Design Doc には、大きな変更の検討の全体を書く

1 ページの ADR に収まらない大きな変更について、背景、目標と非目標、設計案、代替案の比較、影響範囲と移行計画を書く。決定に至るまでの検討全体を 1 か所にまとめ、レビューの土台にするための文書である。命名規則や凍結の手順など、運用の詳細は [design_doc.md](./design_doc.md) を、本文の雛形は [templates/design_doc.md](./templates/design_doc.md) を参照する。

> 例: 「レート制限を全 API へ展開する設計。対象 API の一覧、共通ミドルウェア化する案と API ごとに実装する案の比較、段階的な適用計画」

Design Doc は `docs/` の下に置くが、性質としては変更系であるため、ステータスを Implemented にした時点で凍結する。将来を縛る決定の正本はここには置かず、Approved になった段階で ADR に切り出して相互にリンクする (→ ADR)。確定後の仕様も置かず、reference に書く (→ reference)。凍結後に実装が Design Doc の内容と食い違っても、更新はしない。現在の状態の正本は、あくまで reference と ADR だからである。

## 恒常系 (現在の状態を表し、変更のたびに更新する)

| 置き場所 | ここに書く (正本) | ここには書かない (→ 正しい置き場) |
| --- | --- | --- |
| コードコメント | 局所的な「なぜこの行・このやり方か」、前提・落とし穴 | 全体の仕様・設計判断 (→ docs / ADR) |
| ADR (`docs/adr/`) | 将来を縛る設計判断と、その文脈・却下案・結果 | 使い方 (→ how-to)、変更手順 (→ PR)、受理後は不変 (supersede で更新) |
| docs / reference | 現在の仕様・API・設定の正確な事実 | なぜそうしたか (→ explanation)、手順 (→ how-to) |
| docs / explanation | 設計の背景・仕組み・なぜこうなっているか (自分の言葉で書く) | ADR・Issue・PR への直接リンク (→ 外部公開時にリンク切れになる)、仕様の網羅的な記述 (→ reference)、手順 (→ how-to) |
| docs / how-to | 特定の目的を達成する手順 | 概念の説明 (→ explanation)、網羅仕様 (→ reference) |
| docs / tutorial | 初学者向けの学習導線 | 網羅的なリファレンス (→ reference) |
| README | プロジェクト概要・導入・docs への入口 | 詳細な仕様 (→ docs) |
| CHANGELOG | バージョン間の利用者向け変更点 (Added/Changed/Fixed 等) | 内部実装の詳細 (→ PR / コミット) |

### コードコメントには、コードから読み取れない局所的な意図を書く

局所的な「なぜこの行・このやり方なのか」といった理由や、コードからは読み取れない前提条件、落とし穴を書く。対象は、そのコードを読む人がコード自体からは読み取れない意図に限る。

> 例: 「// トークンの補充は定期実行ではなく参照時の遅延計算で行う。定期実行はプロセス再起動のたびに欠落するため」

全体の仕様や設計に関する判断は書かない (→ docs / ADR)。こうした情報をコメントに書くと、複数のファイルに散らばり、コードの変更とともに陳腐化していく。全体に関わる事柄は docs や ADR に集約し、コメントは局所的な意図に絞る。こうすることで、コメントと docs の境界に隙間や重複が生じないようにする。

### ADR には、将来を縛る設計判断を書く

将来の実装を縛る設計判断を、その背景となる文脈、検討した代替案、結果とともに書く。1 ファイルにつき 1 つの決定とし、`docs/adr/YYYYMMDD-<題>.md` のように作成日を付けて配置する。いつ書くか、ステータスの遷移、supersede (置き換え) の手順については [adr.md](./adr.md) を正本とし、ここでは繰り返さない。

> 例: 「ADR-20260115: レート制限の状態は Redis に置く。プロセス内メモリに持つ案は、多インスタンス構成で状態を共有できないため却下」

使い方 (→ how-to) や、個々の変更手順 (→ PR) は書かない。受理された ADR の本文は編集せず、決定を覆す場合は新しい ADR を作成して古い ADR を supersede する。

### docs の 4 分類は、読者のどの問いに答えるかで書き分ける

docs は、読者のニーズに応じて 4 つ (reference / explanation / how-to / tutorial) に分類する。分類の考え方や書き分けの詳細については [diataxis.md](./diataxis.md) を正本とし、ここでは置き場の判断に必要な範囲と実例のみを示す。同じ「レート制限」を扱う場合でも、読者のどのような問いに答えるかによって置き場が変わることが、実例から読み取れるはずである。

docs の 4 分類から ADR、Design Doc、Issue、PR へはリンクを張らない。docs は外部に公開される可能性があり、外部の読者はこれらの内部リンクにアクセスできないためである。この制約の詳細については [diataxis.md](./diataxis.md) を参照する。

reference には、現在の仕様・API・設定に関する正確な事実を書く。「いまシステムが何をするか」の正本であり、なぜそうしたか (→ explanation) や手順 (→ how-to) は交えない。

> 例: 「アップロード API の上限は 1 ユーザーあたり毎分 60 リクエスト。超過時は 429 と Retry-After ヘッダーを返す」

explanation には、設計の背景・仕組み・なぜ現在の形になっているかを、外部の読者にも通じる自分の言葉で書く。ADR の内容を読者向けに要約して書き直すのは問題ないが、ADR そのものへのリンクは張らない。

> 例: 「なぜ固定ウィンドウ方式ではなくトークンバケット方式なのか。バーストの扱いがどう変わるか」

how-to には、特定の目的を達成するための手順を書く。すでにシステムに慣れた読者が目の前の課題を解決するための道筋であり、概念の説明 (→ explanation) や網羅的な仕様 (→ reference) には踏み込まず、手順に必要な範囲だけを扱う。

> 例: 「特定ユーザーの上限を一時的に引き上げるには」

tutorial には、初学者が手を動かしながら学ぶための導線を書く。正確に網羅すること (→ reference) よりも、学習者を最後までやり遂げさせて成功体験を与えることを優先する。

> 例: 「クライアントを作成して、最初のファイルをアップロードするまで」

### README は、初めて訪れた人のための表紙にする

プロジェクトの概要・導入手順・docs への入口を書く。初めて訪れた人が「何のためのもので、どう使い始めるか」を把握し、より詳しい文書へ進むための表紙である。

> 例: 「本プロジェクトはファイル共有サービスの API サーバーである。`make setup` で開発環境を構築できる。API の仕様は docs/reference/ を参照」

詳細な仕様は書かない (→ docs)。README に詳細を書き込むと docs との二重管理になるうえ、表紙としても読みにくくなってしまう。詳細は docs に委ね、README からリンクを張るにとどめる。

### CHANGELOG には、利用者にとって意味のある変更点を書く

バージョン間における、利用者にとって意味のある変更点を書く。分類・書式・Semantic Versioning との対応については [changelog.md](./changelog.md) を正本とする。

> 例: 「Added: アップロード API にレート制限を導入した (1 ユーザーあたり毎分 60 リクエスト)。超過時は 429 を返す」

内部実装の詳細は書かない (→ PR / コミット)。コミットは開発者が読む記録であり、CHANGELOG は利用者が読む記録であるため、コミットの羅列を貼っても利用者には意味が伝わらない。利用者の言葉に書き直して記載する。
