# 置き場マップの詳細

[SKILL.md](../SKILL.md) の 2 表の各行について,「書くこと」「書かないこと (正しい置き場)」と、その判断根拠を述べる。どこに書くか迷ったときの最終的な拠り所とする。

判断の出発点は常に同じである。その情報が「1 つの変更に紐づく記録 (変更系)」なのか、それとも「現在の状態を表す記録 (恒常系)」なのかをまず見分ける。変更系はマージ後に凍結し、恒常系は変更のたびに更新する。

## 変更系

### Issue 本文

課題・背景・動機、利用者視点での期待する動作、受け入れ基準 (完了条件)、スコープ外を書く。「なぜ必要か」と「何をどうしたいか」を利用者の視点で述べる場所であり、第三者が対応の要否と優先度を判断できる状態にする。

実装手段・API・設定値・アーキテクチャ全体像は書かない。これらはコードや docs から復元でき、Issue に書くと二重管理になって陳腐化する (→ docs)。進捗も書かない (→ Issue コメント)。Issue 本文は作成時に書いたら、以後は原則として書き換えない。

### Issue コメント (実装計画)

着手前の実装方針、変更対象のファイル・モジュール、検討した代替案と却下理由、テスト方針を書く。コードを大量に生成した後で方針の誤りに気づく手戻りを防ぐための、着手前の合意点である。

確定後の「現在の仕様」は書かない (→ reference)。実装の過程で生まれた、将来の実装を縛る決定は ADR へ昇格させる (→ ADR)。実装計画はあくまで着手前の予測であり、結果との差は PR に書く。

### Issue コメント (進捗)

ブランチ名・PR・状態の変化だけを書く。中断・再開時に、いまどの段階にいるかをコメント履歴から復元できるようにする。

設計・仕様・知見は書かない。それぞれの正本に書き、ここには状態の遷移だけを残す。

### PR 本文

対応 Issue、変更の概要 (この変更で何をしたか)、実装計画からの乖離点 (計画コメントへのリンクを添え、変わった点だけを書く)、スコープ外、動作確認・テストの結果、影響を受ける恒常ドキュメントを更新したかの確認を書く。

Issue の背景を再掲しない (Issue へリンクする)。現在仕様の全体は書かない (→ docs)。将来を縛る決定の正本も置かない。決定は ADR に書き、PR からはそこへリンクする。

### コミットメッセージ

1 つの関心事に対する変更内容と、その変更を行った理由を書く。切り戻しやレビューがしやすい単位に保つ。

大きな設計判断の正本は置かない。コミットメッセージは個々の変更の記録であり、横断的な決定は ADR に書いてそこへ委ねる。

## 恒常系

### コードコメント

局所的な「なぜこの行・このやり方なのか」、コードからは読み取れない前提や落とし穴を書く。対象は、その場所を読む人がコードだけでは分からない意図に限る。

全体の仕様や設計判断は書かない。これらはコードと一緒に腐りやすく、複数の場所に散る。全体に関わる事柄は docs や ADR に集約し、コメントは局所的な意図に絞る。こうすることで、コメントと docs の境界に隙間も重複も生じない。

### ADR (`docs/adr`)

将来の実装を縛る設計判断と、その文脈・検討した代替案・結果を書く。1 ファイルにつき 1 つの決定とし、`docs/adr/NNNN-<題>.md` に連番で置く。詳しい運用は [adr.md](./adr.md) を参照する。

使い方 (→ how-to) や変更の手順 (→ PR) は書かない。受理した ADR は編集せず、決定を覆すときは新しい ADR を書いて旧 ADR を supersede する。これにより、決定の正本が常に 1 つに保たれ、過去の判断根拠も失われない。

### docs / reference

現在の仕様・API・設定の正確な事実を書く。「いまシステムが何をするか」の正本であり、解釈や手順を交えず、正確さと網羅性を優先する。

「なぜそうしたか」は書かない (→ explanation / ADR)。特定の目的を達成する手順も書かない (→ how-to)。

### docs / explanation

設計の背景・仕組み・なぜいまの形になっているかを書く。読者の理解を深めるための説明であり、設計の文脈や経緯を扱う。

網羅的な仕様の逐一は書かない (→ reference)。手順も書かない (→ how-to)。

### docs / how-to

特定の目的を達成するための手順を書く。すでにある程度慣れた読者が、目の前の課題を解決するための実践的な道筋を示す。

概念の説明は書かない (→ explanation)。網羅的な仕様も書かない (→ reference)。

### docs / tutorial

初学者が手を動かしながら学ぶための学習導線を書く。何かを達成させることより、学習者を導いて成功体験を与えることを目的とする。

網羅的なリファレンスは書かない (→ reference)。4 分類の使い分けは [diataxis.md](./diataxis.md) を参照する。

### README

プロジェクトの概要・導入手順・docs への入口を書く。初めて訪れた人が「何のためのもので、どう使い始めるか」を把握し、より詳しい文書へ進めるようにする。

詳細な仕様は書かない (→ docs)。README は入口であり、詳細は docs へ委ねてリンクする。

### CHANGELOG

バージョン間の、利用者にとって意味のある変更点を書く。利用者の言葉で、リリースごとに分類して記録する。詳しい運用は [changelog.md](./changelog.md) を参照する。

内部実装の詳細は書かない (→ PR / コミット)。コミットの羅列をそのまま貼らず、利用者目線で書き直す。
