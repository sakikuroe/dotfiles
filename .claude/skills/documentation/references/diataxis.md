# docs の 4 分類 (Diátaxis)

docs は、読者の必要に応じて性質の異なる 4 種類に分かれる。これは Diátaxis (https://diataxis.fr/) の考え方に基づく。4 つを 1 つの文書に混ぜると、学びたい人には回り道が多く、事実を確認したい人には説明が邪魔な、どの読者にとっても中途半端な文書になる。そこで役割ごとにディレクトリーを分け、それぞれを 1 つの目的に絞って書く。

分類は 2 つの軸で整理できる。1 つは「学習のため (study) か、仕事のため (work) か」、もう 1 つは「実践 (action) か、知識 (cognition) か」である。

| 分類 | 目的 | 読者の状態 | 性質 |
| --- | --- | --- | --- |
| tutorial | 学習させる | これから学ぶ初学者 | 学習 × 実践 |
| how-to | 課題を解決させる | すでに慣れた利用者 | 仕事 × 実践 |
| reference | 正確な事実を提供する | 作業中に確認したい利用者 | 仕事 × 知識 |
| explanation | 理解を深めさせる | 背景を知りたい読者 | 学習 × 知識 |

## tutorial (学習)

初学者が手を動かしながら学び、成功体験を得るための導線を書く。書き手が責任を持って学習者を導く文書であり、正確に網羅することよりも、最後までやり遂げさせることを優先する。途中で選択肢や例外を並べ立てると学習者は迷子になるため、網羅的な仕様や細かい選択肢の説明は書かない (→ reference)。

## how-to (目的達成の手順)

すでにある程度慣れた利用者が、具体的な目的を達成するための手順を書く。「〜するには」という形で、現実の課題に対する実践的な道筋を示す。読者は目的を果たしたいのであって学びたいわけではないため、概念の説明 (→ explanation) や網羅的な仕様 (→ reference) には踏み込まず、手順に必要な範囲だけを扱う。

## reference (現在の仕様)

現在の仕様・API・設定の正確な事実を書く。「いまシステムが何をするか」の正本であり、解釈や手順を交えず、正確さと網羅性を優先する。コードから生成できるものは生成し、手で書く部分は現在の状態に追従させる。なぜそうなっているか (→ explanation / ADR) や、手順 (→ how-to) は書かない。事実の確認に来た読者にとって、それらは探しものを埋もれさせる雑音になるためである。

## explanation (背景・仕組み)

設計の背景・仕組み・なぜいまの形になっているかを書く。読者の理解を深めることが目的であり、設計の文脈や経緯、検討の幅を扱う。将来を縛る個別の意思決定を扱うときも、ADR や Design Doc へは直接リンクせず (後述の「リンクは一方通行にする」を参照)、決定の内容と「なぜ今この形なのか」を読者に通じる言葉で書き直す。決定の詳しい経緯や却下した代替案まで持ち込む必要はなく、読者に必要なのは現状への納得だけである。仕様の網羅的な記述 (→ reference) や手順 (→ how-to) は書かない。

## リンクは一方通行にする

docs (`docs/book/`) は、将来ビルドして外部に公開する可能性がある。外部の読者は ADR・Design Doc・Issue・PR にアクセスできないため、docs からこれらへリンクすると、公開した瞬間に壊れたリンクだらけになる。

そこで、リンクの向きを一方通行に固定する。ADR・Design Doc・Issue・PR から docs へリンクするのは自由だが、docs からこれらへリンクしてはならない。docs はそれだけで読んで完結する世界として書く。設計の理由に触れたい場合も、ADR の中身を要約し読者向けの言葉で書き直すのであって、ADR そのものへのリンクではない。

## ディレクトリー構成

docs/book/ 配下は、将来ビルドして外部に公開することを想定した構成にするが、実際にビルドする仕組みを導入するかどうかは各プロジェクトの判断に委ねる。導入しない場合も、このディレクトリー構成自体は変えない。

```
docs/
├── adr/            # 将来を縛る設計判断 (adr.md を参照)
│   └── 20260115-rate-limit-state-in-redis.md
├── design/         # 大きな変更の設計提案。変更系であり、確定後は凍結する (design_doc.md を参照)
│   └── 20260110-rate-limit-rollout.md
└── book/           # Diátaxis の 4 分類。外部公開を想定するため docs/adr, docs/design とは分離する
    └── src/
        ├── SUMMARY.md      # 目次の正本。ページを追加したら忘れずに 1 行足す
        ├── tutorial/       # 学習導線
        │   └── getting-started.md
        ├── how-to/         # 目的達成の手順
        │   └── raise-a-users-rate-limit.md
        ├── reference/      # 現在の仕様・API・設定
        │   └── api.md
        └── explanation/    # 設計の背景・仕組み
            └── why-token-bucket.md
```

adr/ と design/ は Diátaxis の 4 分類の外にあり、かつ外部に公開しない前提の文書であるため、book/ の外に置く。adr/ の運用は [adr.md](./adr.md) に、design/ の運用は [design_doc.md](./design_doc.md) に従う。ビルドする仕組みを実際に導入するかどうかは各プロジェクトの判断に委ね、本スキルはその導入自体を必須にはしない。

小さなプロジェクトでは、すべての分類を最初から埋める必要はなく、必要になった分類から作ればよい。ただし、文書を書くときは「これはどの分類か」を常に意識し、1 つの文書に複数の分類を混ぜないようにする。
