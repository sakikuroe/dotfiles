# docs の 4 分類 (Diátaxis)

docs は, 読者の必要に応じて性質の異なる 4 種類に分かれる. これは Diátaxis (https://diataxis.fr/) の考え方に基づく. 4 つを混ぜると, どの読者にとっても中途半端な文書になる. 役割ごとにディレクトリーを分け, それぞれを 1 つの目的に絞って書く.

2 つの軸で整理できる. 1 つは「学習のため (study) か, 仕事のため (work) か」, もう 1 つは「実践 (action) か, 知識 (cognition) か」である.

| 分類 | 目的 | 読者の状態 | 性質 |
| --- | --- | --- | --- |
| tutorial | 学習させる | これから学ぶ初学者 | 学習 × 実践 |
| how-to | 課題を解決させる | すでに慣れた利用者 | 仕事 × 実践 |
| reference | 正確な事実を提供する | 作業中に確認したい利用者 | 仕事 × 知識 |
| explanation | 理解を深めさせる | 背景を知りたい読者 | 学習 × 知識 |

## tutorial (学習)

初学者が手を動かしながら学び, 成功体験を得るための導線を書く. 書き手が責任を持って学習者を導く. 何かを正確に網羅することより, 最後までやり遂げさせることを優先する.

網羅的な仕様や, 細かい選択肢の説明は書かない (→ reference).

## how-to (目的達成の手順)

すでにある程度慣れた利用者が, 具体的な目的を達成するための手順を書く. 「〜するには」という形で, 現実の課題に対する実践的な道筋を示す.

概念の説明 (→ explanation) や, 網羅的な仕様 (→ reference) は書かない. 手順に必要な範囲だけを扱う.

## reference (現在の仕様)

現在の仕様・API・設定の正確な事実を書く. 「いまシステムが何をするか」の正本であり, 解釈や手順を交えず, 正確さと網羅性を優先する. コードから生成できるものは生成し, 手で書く部分は現在の状態に追従させる.

なぜそうなっているか (→ explanation / ADR) や, 手順 (→ how-to) は書かない.

## explanation (背景・仕組み)

設計の背景・仕組み・なぜいまの形になっているかを書く. 読者の理解を深めることが目的で, 設計の文脈や経緯, 検討の幅を扱う. ただし, 将来を縛る個別の意思決定そのものは ADR を正本とし, ここからはリンクする.

網羅的な仕様の逐一 (→ reference) や, 手順 (→ how-to) は書かない.

## ディレクトリー構成

```
docs/
├── adr/            # 将来を縛る設計判断 (ADR. adr.md を参照)
├── tutorial/       # 学習導線
├── how-to/         # 目的達成の手順
├── reference/      # 現在の仕様・API・設定
└── explanation/    # 設計の背景・仕組み
```

小さなプロジェクトでは, すべての分類を最初から埋める必要はない. 必要になった分類から作る. ただし, 文書を書くときは「これはどの分類か」を意識し, 1 つの文書に複数の分類を混ぜないようにする.
