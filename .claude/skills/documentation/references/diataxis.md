# docs の 4 分類 (Diátaxis)

docs は、読者の必要に応じて性質の異なる 4 種類に分かれる。これは Diátaxis (https://diataxis.fr/) の考え方に基づく。4 つを 1 つの文書に混ぜると、学びたい人には回り道が多く、事実を確認したい人には説明が邪魔な、どの読者にとっても中途半端な文書になる。そこで役割ごとにディレクトリーを分け、それぞれを 1 つの目的に絞って書く。

分類は 2 つの軸で整理できる。1 つは「学習のため (study) か、仕事のため (work) か」、もう 1 つは「実践 (action) か、知識 (cognition) か」である。

| 分類 | 目的 | 読者の状態 | 性質 |
| --- | --- | --- | --- |
| tutorial | 学習させる | これから学ぶ初学者 | 学習 × 実践 |
| how-to | 課題を解決させる | すでに慣れた利用者 | 仕事 × 実践 |
| reference | 正確な事実を提供する | 作業中に確認したい利用者 | 仕事 × 知識 |
| explanation | 理解を深めさせる | 背景を知りたい読者 | 学習 × 知識 |

各分類のテンプレートは [templates/](./templates/) に置いてある。

## tutorial (学習)

初学者が手を動かしながら学び、成功体験を得るための導線を書く。書き手が責任を持って学習者を導く文書であり、正確に網羅することよりも、最後までやり遂げさせることを優先する。道は一本に絞り、各ステップの終わりに結果を確認できる一文を入れる。途中で選択肢や例外を並べ立てると学習者は迷子になるため、網羅的な仕様や細かい選択肢の説明は書かない (→ reference)。説明したくなったら一言にとどめ、詳しい理屈は explanation へのドキュメント内リンクに逃す。

## how-to (目的達成の手順)

すでにある程度慣れた利用者が、具体的な目的を達成するための手順を書く。「〜するには」という形で、現実の課題に対する実践的な道筋を示す。読者は目的を果たしたいのであって学びたいわけではないため、概念の説明 (→ explanation) や網羅的な仕様 (→ reference) には踏み込まず、手順に必要な範囲だけを扱う。現実の条件による分岐は書いてよいが、全選択肢の網羅はしない。

## reference (現在の仕様)

現在の仕様・API・設定の正確な事実を書く。「いまシステムが何をするか」の正本であり、解釈や手順を交えず、正確さと網羅性を優先する。構成は製品そのものの構造 (設定画面のタブ構成、API のエンドポイント構成など) に合わせ、すべての項目を同じ型で書く。コードから生成できるものは生成し、手で書く部分は現在の状態に追従させる。なぜそうなっているか (→ explanation) や、手順 (→ how-to) は書かない。事実の確認に来た読者にとって、それらは探しものを埋もれさせる雑音になるためである。

## explanation (背景・仕組み)

設計の背景・仕組み・なぜいまの形になっているかを書く。読者の理解を深めることが目的であり、設計の文脈や経緯、検討の幅を扱う。4 分類の中で唯一、複数の視点や意見を書いてよい場所であるが、意見は意見だと分かる形で書く。仕様の網羅的な記述 (→ reference) や手順 (→ how-to) は書かない。

explanation と ADR は「なぜ」を扱う点で似ているが、読者と目的が異なる。ADR はチーム内部の意思決定の記録であり、検討した選択肢や却下理由を正確に残すことが目的である。explanation はドキュメントの読者 (外部を含む) に向けた読み物であり、「なぜ現在の形になっているか」を読者に通じる言葉で納得させることが目的である。したがって explanation には ADR の内容を読者向けに要約して書き直してよいが、ADR への直接のリンクは張らない (後述のリンク規則を参照)。

## docs から ADR・Design Doc・Issue・PR へリンクしない

docs の 4 分類 (tutorial / how-to / reference / explanation) から、ADR、Design Doc、Issue、PR へのリンクは張らない。逆に、ADR・Design Doc・Issue・PR から docs へのリンクは自由に張ってよい。

この規則の理由は、docs を外部に公開できる状態に保つためである。docs は静的サイトジェネレーター (mdBook、Sphinx、VitePress など) でビルドして外部に公開する場合がある。外部の読者は ADR や Issue にアクセスできないため、docs からこれらへリンクがあると、公開した時点でリンク切れになる。公開しない運用であっても、最初からこの規則に従って書いておけば、公開が必要になったときに全ページを点検して書き直す作業が発生しない。

explanation で設計の理由に触れたいときは、ADR へリンクする代わりに、外部の読者にも通じる言葉でその理由を自分の言葉として書く。ADR の内容を要約して書き直すのは問題ない (正本の原則が防ぎたいのは無自覚なコピーの放置であり、読者に合わせた書き直しは正当である)。

## ディレクトリー構成

```
docs/
├── adr/            # 将来を縛る設計判断 (adr.md を参照)
│   └── 20260115-rate-limit-state-in-redis.md
├── design/         # 大きな変更の設計提案。変更系であり、確定後は凍結する (design_doc.md を参照)
│   └── 20260110-rate-limit-rollout.md
├── tutorial/       # 学習導線
│   └── getting-started.md
├── how-to/         # 目的達成の手順
│   └── raise-a-users-rate-limit.md
├── reference/      # 現在の仕様・API・設定
│   └── api.md
└── explanation/    # 設計の背景・仕組み
    └── why-token-bucket.md
```

adr/ と design/ は Diátaxis の 4 分類の外にあるが、置き場としては docs の下に同居させる。adr/ の運用は [adr.md](./adr.md) に、design/ の運用は [design_doc.md](./design_doc.md) に従う。

小さなプロジェクトでは、すべての分類を最初から埋める必要はなく、必要になった分類から作ればよい。まず書くのは how-to と reference である。いま困っている読者に直接効き、機能追加の PR のついでに書ける粒度だからである。tutorial は初めて触る人が現れる段階で、explanation は同じ「なぜ」を 2 回聞かれたら書く、という目安で増やす。ただし、文書を書くときは「これはどの分類か」を常に意識し、1 つの文書に複数の分類を混ぜないようにする。
