# docs の 4 分類 (Diátaxis)

docs は、読者のニーズに応じて性質が異なる 4 種類に分類される。これは、Diátaxis (https://diataxis.fr/) の考え方に基づく。これら 4 つを 1 つの文書に混在させると、これから学びたい人には回り道が多くなり、事実を確認したい人には余計な説明が邪魔になるなど、どの読者にとっても中途半端なものになってしまう。そこで、役割ごとにディレクトリーを分け、それぞれ単一の目的に絞って文書を作成する。

これらの分類は 2 つの軸で整理できる。1 つは「学習目的 (study) か、実務目的 (work) か」、もう 1 つは「実践 (action) か、知識 (cognition) か」である。

| 分類 | 目的 | 読者の状態 | 性質 |
| --- | --- | --- | --- |
| tutorial | 学習させる | これから学ぶ初学者 | 学習 × 実践 |
| how-to | 課題を解決させる | すでに慣れた利用者 | 仕事 × 実践 |
| reference | 正確な事実を提供する | 作業中に確認したい利用者 | 仕事 × 知識 |
| explanation | 理解を深めさせる | 背景を知りたい読者 | 学習 × 知識 |

各分類のテンプレートは [templates/](./templates/) に用意してある。

## tutorial (学習)

初学者が実際に手を動かしながら学び、成功体験を得られるようにするための導線を書く。書き手が責任を持って学習者を導くための文書であり、情報を正確に網羅することよりも、読者に最後までやり遂げさせることを優先する。手順は一本道に絞り、各ステップの最後には結果を確認できる一文を入れる。途中で選択肢や例外を並べ立てると学習者が迷子になってしまうため、網羅的な仕様や細かい選択肢の説明は書かない (→ reference)。説明を入れる場合は一言にとどめ、詳しい理屈は explanation へのドキュメント内リンクを張ってそちらに委ねる。

## how-to (目的達成の手順)

すでにある程度操作に慣れた利用者が、具体的な目的を達成するための手順を書く。「〜するには」という形式で、現実の課題に対する実践的な道筋を示す。ここでの読者は目的を達成したいのであって、基礎から学びたいわけではない。そのため、概念の説明 (→ explanation) や網羅的な仕様 (→ reference) には踏み込まず、手順に必要な範囲の情報だけを扱う。実際の条件に応じた分岐を書いてもよいが、すべての選択肢を網羅する必要はない。

## reference (現在の仕様)

現在の仕様・API・設定に関する正確な事実を書く。「現在のシステムがどう振る舞うか」を示す正本であり、独自の解釈や操作手順は交えず、正確さと網羅性を最優先する。ドキュメントの構成は製品自体の構造 (設定画面のタブ構成や API のエンドポイント構成など) に合わせ、すべての項目を同じ形式で記述する。コードから自動生成できる部分は生成に任せ、手動で書く部分は常に現在のシステムの状態に追従させる。なぜそのような仕様になっているのか (→ explanation) や、具体的な操作手順 (→ how-to) は書かない。事実を確認したい読者にとって、それらの情報は本当に必要な探しものを埋もれさせるノイズになってしまうからである。

## explanation (背景・仕組み)

設計の背景や仕組み、なぜ現在の形になったのかを書く。読者の理解を深めることが目的であり、設計の文脈や経緯、検討した選択肢の幅などを記載する。4 つの分類の中で唯一、複数の視点や意見を交えてもよい場所だが、意見を書く際はそれが個人の見解であると明確に分かるように書く。ここには、仕様の網羅的な説明 (→ reference) や具体的な操作手順 (→ how-to) は書かない。

explanation と ADR はいずれも「なぜ」を扱う点では共通しているが、想定する読者と目的が異なる。ADR は開発チーム内部における意思決定の記録であり、検討した選択肢や却下した理由を正確に残すことが目的である。一方、explanation はドキュメントの読者 (外部の利用者を含む) に向けた読み物であり、「なぜ現在の形になっているのか」を読者に伝わる言葉で説明し、納得してもらうことが目的である。したがって、explanation 内で ADR の内容を読者向けに要約して書き直すのは構わないが、ADR への直接リンクは張らないようにする (後述のリンク規則を参照)。

## docs から ADR・Design Doc・Issue・PR へリンクしない

docs の 4 分類 (tutorial / how-to / reference / explanation) からは、ADR、Design Doc、Issue、PR へのリンクを張らないようにする。逆に、ADR や Design Doc などから docs の 4 分類へのリンクは自由に張って構わない。

この規則を設ける理由は、docs をいつでも外部へ公開できる状態に保つためである。docs は静的サイトジェネレーター (mdBook、Sphinx、VitePress など) でビルドし、外部に公開する可能性がある。外部の読者はリポジトリ内の ADR や Issue にはアクセスできないため、docs からリンクを張ってしまうと、公開した時点でリンク切れになってしまう。当面は公開しない運用であっても、最初からこの規則に従って書いておけば、将来的に公開が必要になった際に全ページを点検して書き直す手間を省くことができる。

explanation で設計の理由に触れたい場合は、ADR へ直接リンクするのではなく、外部の読者にも伝わる言葉を用いて、自分の言葉でその理由を記述する。その際、ADR の内容を要約して書き直すことは全く問題ない (正本の原則が防ぎたいのは無自覚なコピーの放置であり、読者に合わせた適切な書き直しは正当な手続きである)。

## ディレクトリー構成

```
docs/
├── adr/            # 将来を縛る設計判断 (adr.md を参照)
│   └── 20260115-rate-limit-state-in-redis.md
├── design/         # 大きな変更の設計提案。変更系であり、確定後は凍結する (design_doc.md を参照)
│   └── 20260110-rate-limit-rollout.md
├── tutorial/       # 学習導線
│   └── getting-started.md
├── how-to/         # 目的達成の手順
│   └── raise-a-users-rate-limit.md
├── reference/      # 現在の仕様・API・設定
│   └── api.md
└── explanation/    # 設計の背景・仕組み
    └── why-token-bucket.md
```

adr/ と design/ は Diátaxis の 4 分類には含まれないが、ディレクトリーの置き場としては docs の配下に同居させる。adr/ の運用は [adr.md](./adr.md) に、design/ の運用は [design_doc.md](./design_doc.md) に従うものとする。

小さなプロジェクトの場合、最初からすべての分類を埋める必要はなく、必要になったものから順に作成していけばよい。真っ先に書くべきなのは how-to と reference である。これらは現在困っている読者の助けに直結し、機能追加の PR のついでに書きやすい粒度だからである。tutorial はプロジェクトに初めて触れる人が現れた段階で、explanation は同じ「なぜ」を 2 回聞かれたタイミングで追加する、といった目安で運用するとよい。ただし、文書を書く際は常に「これはどの分類か」を意識し、1 つの文書に複数の分類を混ぜないよう注意すること。
