# ADR-NNNN: タイトル

<!--
ADR は、将来の実装を縛る設計判断を 1 つだけ記録する文書である。「なぜその選択をし、なぜ他を退けたか」を、将来の自分やチームが参照できるようにする。いつ書くか、ファイルの命名、レビューの流れ、受理後の supersede といった運用は documentation スキルの `references/adr.md` を正本とするため、ここでは繰り返さない。運用に迷ったらそちらを読むこと。

タイトルは、何を決めたかがタイトルだけで分かる句で書く。「〜を採用する」「〜に移行する」「〜を廃止する」のような形が望ましい。「データベースについて」のように対象しか書かないタイトルは、決定内容が伝わらないため不適切である。

各節の HTML コメントは執筆時の手引きであり、書き上げたら削除する。
-->

## 本決定が必要になった背景と状況

<!--
この判断が必要になった背景と状況を、価値判断を交えず事実と制約条件として書く。結論や主張はここには含めない。ここに結論を書いてしまうと、後続の「検討した選択肢」を読者が公平に評価できなくなるためである。読み手がこの節だけで「なぜ判断が必要なのか」を理解できる粒度で書き、関連する Issue があればリンクする。

適切な例:

  ```md
  注文管理システムのデータベースを選定する必要がある。
  想定データ量は年間 100 万件の注文レコードであり、今後 3 年で 5 倍程度の増加を見込む。
  注文のメタデータには可変項目 (配送先の追加属性、キャンペーン情報等) が含まれるため、
  これらを柔軟にクエリできる必要がある。
  一方、注文と在庫の整合性を保つためにトランザクション制御は必須である。
  チームには MySQL の運用経験が 3 年以上あるメンバーが 4 名いるが、
  他の DBMS の実務経験は限定的である。
  ```

  技術的な要件 (データ量、可変項目のクエリ、トランザクション) と組織的な制約 (チームのスキルセット) の両方が客観的に書かれており、読み手は後続の「検討した選択肢」を評価する判断材料をここから得られる。

不適切な例:

  ```md
  PostgreSQL が良さそうであるため採用したい。
  ```

  結論が先に述べられており、背景も制約条件も書かれていない。この節には、判断が必要になった状況だけを書くこと。
-->

## 検討した選択肢とそれぞれの長所・短所

<!--
検討した選択肢を 2 つ以上挙げ、それぞれの長所と短所の両方を書く。「何もしない (現状維持を続ける)」を常に選択肢に含める。何もしないことが明らかに不適切であっても、なぜ不適切なのかを明示することで判断の基準線が定まるためである。同様に、ある選択肢が明らかに最善であっても短所 (トレードオフ) を明記しておく。それが将来この決定を見直すときの判断材料になる。

各選択肢の見出しは、「○○を採用する案」のように案の内容が分かる句で書く。

適切な例:

  ```md
  ### PostgreSQL を採用する案

  - 長所: JSONB によりスキーマレスなデータと構造化データを 1 つの DB で扱うことができる。
    GIN インデックスにより JSONB カラムの検索性能も確保できる。
    エコシステムが成熟しており、マイグレーションツールや監視ツールの選択肢が豊富。
  - 短所: チームに PostgreSQL の実務経験が少なく、MySQL との差分の学習コストが発生する。
    見積もりでは 1〜2 週間程度である。

  ### MySQL を採用する案

  - 長所: チームに 3 年以上の運用経験があり、即座に開発・運用を開始できる。
  - 短所: JSON 型のインデックス性能が PostgreSQL の JSONB に比べて劣る。
    可変メタデータのクエリが頻繁に発生するユースケースでは、
    パフォーマンスの問題が将来的に顕在化するリスクがある。

  ### MongoDB を採用する案

  - 長所: スキーマレスで柔軟。可変メタデータの格納・クエリに最も自然に適合する。
  - 短所: 注文と在庫の整合性を保つためのトランザクション制御が RDB に比べて制限される。
    マルチドキュメントトランザクションは対応しているが、性能と運用の複雑さが増す。

  ### 何もしない (判断を先送りする)

  - 短所: データベースが未決定のまま開発を進められないため、先送りはできない。
  ```

  各選択肢に長所と短所の両面が書かれており、背景の節で挙げた要件 (JSONB クエリ、トランザクション、チームスキル) に対して各選択肢がどう応えるかが具体的に示されている。

不適切な例:

  ```md
  PostgreSQL か MySQL で迷った。
  ```

  選択肢が列挙されておらず、それぞれの長所・短所も書かれていないため、判断の根拠が読み取れない。
-->

## 最終的に採用する方針

<!--
最終的に採用した選択肢を、「○○を採用する」のように能動態・現在形の 1〜2 文で断定的に書く。

適切な例:

  ```md
  PostgreSQL を採用する。
  ```

  簡潔であり、何を決定したかが一文で分かる。

不適切な例:

  ```md
  PostgreSQL が良さそうなので使おうと思う。
  ```

  「良さそう」「思う」は決定ではなく感想である。決定は断定的に書くこと。
-->

## その方針を採用し他を退けた理由

<!--
なぜその選択肢を採用し、なぜ他を採用しなかったかを書く。「検討した選択肢」の節が客観的な事実の列挙であるのに対し、この節はそれらを踏まえた判断の説明である。「この文脈ではなぜこの選択肢が最善と判断したか」を、背景の節で挙げた要件に紐づけて述べる。

適切な例:

  ```md
  JSONB のネイティブ対応により、可変メタデータと構造化データを
  1 つの DB で扱えることが最大の決め手である。MySQL の JSON 型は
  インデックス性能で劣り、メタデータの検索頻度を考慮すると
  将来的なパフォーマンスリスクが高い。

  MongoDB はスキーマの柔軟性では最も優れるが、注文と在庫の整合性を
  トランザクションで厳密に保証する必要がある本システムでは、
  RDB と比較して運用の複雑さが増す。

  チームの学習コスト (MySQL との差分、1〜2 週間と見積もり) は許容範囲と判断した。
  PostgreSQL と MySQL は SQL としての共通部分が大きく、差分は設定・運用面が中心であるため、
  実開発と並行して習得可能と考える。
  ```

  採用した選択肢の決定的な強みと、他の選択肢を退けた具体的な理由が、背景の要件に紐づいて説明されている。

不適切な例:

  ```md
  PostgreSQL が一番良いと思ったから。
  ```

  なぜ良いと判断したかが書かれておらず、「検討した選択肢」の情報を踏まえた説明になっていない。
-->

## 本決定によって生じる影響と後続の作業

<!--
この決定によって生じる影響を、良い面だけでなく、受け入れたトレードオフ (負の影響) も含めて書く。この決定に伴って発生する後続の作業 (Issue、タスク、設定変更等) があれば、Issue 番号つきで具体的に挙げる。番号を添えておくことで、対応漏れを防止できる。

適切な例:

  ```md
  - マイグレーションツールとして node-pg-migrate を導入する。
  - JSONB カラムには GIN インデックスを設計する必要がある (#234 で対応)。
  - MySQL 経験者向けの差分ガイドを `docs/how-to/` に追加する (#235 で対応)。
  - PostgreSQL の運用監視は既存の Datadog で対応できる (追加設定が必要)。
  - トレードオフ: チーム内に PostgreSQL の深い運用知識を持つメンバーが
    おらず、初期は障害対応に時間がかかる可能性がある。
  ```

不適切な例:

  ```md
  PostgreSQL を使い始める。
  ```

  決定の繰り返しであり、結果として何が起きるかが書かれていない。
-->
