# ADR-NNNN: タイトル

<!--
Architecture Decision Record（ADR）は、設計上の重要な判断を 1 つだけ記録する文書である。
「なぜその選択をし、なぜ他を選ばなかったか」を将来の自分やチームメンバーが参照できるようにすることが目的である。

ADR を書くかどうかの判断基準:
  その判断について 6〜12 か月後にまた議論になりそうであれば、ADR を書く。
  例: 技術選定、アーキテクチャ構成の変更、認証方式の選択、データモデルの設計方針など。
  例外: バグ修正、軽微な機能追加、リファクタリングなど判断が自明な変更では ADR は不要。

ADR の運用ルール:
  - 1 ファイル 1 決定。複数の決定を含めない。
  - 1 ページ以内に収める。収まらない場合は Design Doc（docs/design/）として書き、
    その中の判断を ADR として切り出す。
  - ファイル名は `NNNN-判断内容を表すスラッグ.md`（例: `0001-postgresql-を採用する.md`）。
    番号は連番とし、欠番があっても再利用しない。
  - ADR は PR として提出し、レビュー・Approve を経て Accepted とする。
    Accepted になった ADR は不変（immutable）として扱い、結論や理由を事後に編集しない。
    誤字の修正やリンク切れの補修は許容する。
  - 決定を覆す場合は、新しい ADR を作成し、旧 ADR のステータスを `Superseded by ADR-XXXX` に変更する。

タイトルの書き方:
  決定内容を表す名詞句で書く。「〜を採用する」「〜に移行する」「〜を廃止する」のように、
  何を決めたかがタイトルだけで分かるようにする。
  不適切な例: 「データベースについて」（何を決めたか分からない）
-->

## 決定の状態

<!--
以下のいずれかを記載する。日付も併記すること。

  - Proposed（YYYY-MM-DD） — レビュー中。まだ決定は確定していない。
  - Accepted（YYYY-MM-DD） — 決定が確定し、有効である。
  - Deprecated（YYYY-MM-DD） — 非推奨になった。後継はないがもう推奨しない。
  - Superseded by ADR-XXXX（YYYY-MM-DD） — 別の ADR によって置き換えられた。

例: Accepted（2026-06-28）
例: Superseded by ADR-0008（2026-09-15）
-->

## 本決定が必要になった背景と状況

<!--
本セクションには、この判断が必要になった背景と状況を記述する。

事実と制約条件を価値中立に記述すること。結論や主張はここには含めない。
読み手がこのセクションだけで「なぜ判断が必要なのか」を理解できる粒度で書く。
関連する Issue がある場合はリンクする。

以下の例を参考にすること。

適切な例:

  ```md
  注文管理システムのデータベースを選定する必要がある。
  想定データ量は年間 100 万件の注文レコードであり、今後 3 年で 5 倍程度の増加を見込む。
  注文のメタデータには可変項目（配送先の追加属性、キャンペーン情報等）が含まれるため、
  これらを柔軟にクエリできる必要がある。
  一方、注文と在庫の整合性を保つためにトランザクション制御は必須である。
  チームには MySQL の運用経験が 3 年以上あるメンバーが 4 名いるが、
  他の DBMS の実務経験は限定的である。
  ```

  この例では、技術的な要件（データ量、可変項目のクエリ、トランザクション）と
  組織的な制約（チームのスキルセット）の両方が客観的に記述されている。
  読み手は後続の「検討した選択肢」セクションを評価するための判断材料をここから得られる。

不適切な例:

  ```md
  PostgreSQL が良さそうなので採用したい。
  ```

  結論が先に述べられており、背景も制約条件も記述されていない。
  コンテキストには結論を含めず、判断が必要になった状況だけを記述すること。
-->

## 検討した選択肢とそれぞれの長所・短所

<!--
本セクションには、検討した選択肢を列挙し、それぞれの長所と短所を記述する。

選択肢は最低 2 つ以上記載する。「何もしない（現状維持）」は常に選択肢に含めること。
何もしないことが明らかに不適切であっても、なぜ不適切なのかを明示することで
判断の基準線が明確になる。

各選択肢には長所と短所の両方を記述する。ある選択肢が明らかに最善であっても、
その短所（トレードオフ）を明記することで、将来見直す際の判断材料になる。

以下の例を参考にすること。

適切な例:

  ```md
  ### 選択肢 1: PostgreSQL

  - 長所: JSONB によりスキーマレスなデータと構造化データを 1 つの DB で扱える。
    GIN インデックスにより JSONB カラムの検索性能も確保できる。
    エコシステムが成熟しており、マイグレーションツールや監視ツールの選択肢が豊富。
  - 短所: チームに PostgreSQL の実務経験が少なく、MySQL との差分の学習コストが発生する。
    見積もりでは 1〜2 週間程度。

  ### 選択肢 2: MySQL

  - 長所: チームに 3 年以上の運用経験があり、即座に開発・運用を開始できる。
  - 短所: JSON 型のインデックス性能が PostgreSQL の JSONB に比べて劣る。
    可変メタデータのクエリが頻繁に発生するユースケースでは、
    パフォーマンスの問題が将来的に顕在化するリスクがある。

  ### 選択肢 3: MongoDB

  - 長所: スキーマレスで柔軟。可変メタデータの格納・クエリに最も自然に適合する。
  - 短所: 注文と在庫の整合性を保つためのトランザクション制御が RDB に比べて制限される。
    マルチドキュメントトランザクションは対応しているが、性能と運用の複雑さが増す。

  ### 選択肢 4: 何もしない（判断を先送りする）

  - 短所: データベースが未決定のまま開発を進められないため、先送りは不可。
  ```

  各選択肢に長所と短所の両面が記述されている。コンテキストで挙げた要件（JSONB クエリ、
  トランザクション、チームスキル）に対して各選択肢がどう応えるかが具体的に示されている。

不適切な例:

  ```md
  PostgreSQL か MySQL で迷った。
  ```

  選択肢が列挙されておらず、それぞれの長所・短所も記述されていない。
  判断の根拠が読み取れない。
-->

## 採用する方針

<!--
本セクションには、最終的に採用した選択肢を 1〜2 文で記述する。
「○○を採用する」のように、能動態・現在形で明確に書く。

以下の例を参考にすること。

適切な例:

  ```md
  PostgreSQL を採用する。
  ```

  簡潔であり、何を決定したかが一文で分かる。

不適切な例:

  ```md
  PostgreSQL が良さそうなので使おうと思う。
  ```

  「良さそう」「思う」は決定ではなく感想である。決定は断定的に書くこと。
-->

## その方針を採用し他を退けた理由

<!--
本セクションには、なぜその選択肢を採用し、なぜ他の選択肢を採用しなかったかを記述する。

「検討した選択肢」セクションで各選択肢の長所・短所を列挙しているが、
本セクションではそれらを踏まえて「この文脈ではなぜこの選択肢が最善と判断したか」を
統合的に説明する。選択肢セクションが客観的な事実の列挙であるのに対し、
本セクションは判断の根拠の説明である。

以下の例を参考にすること。

適切な例:

  ```md
  JSONB のネイティブ対応により、可変メタデータと構造化データを
  1 つの DB で扱えることが最大の決め手である。MySQL の JSON 型は
  インデックス性能で劣り、メタデータの検索頻度を考慮すると
  将来的なパフォーマンスリスクが高い。

  MongoDB はスキーマの柔軟性では最も優れるが、注文と在庫の整合性を
  トランザクションで厳密に保証する必要がある本システムでは、
  RDB と比較して運用の複雑さが増す。

  チームの学習コスト（MySQL との差分、1〜2 週間と見積もり）は許容範囲と判断した。
  PostgreSQL と MySQL は SQL としての共通部分が大きく、差分は設定・運用面が中心であるため、
  実開発と並行して習得可能と考える。
  ```

  採用した選択肢の決定的な強みと、他の選択肢を退けた具体的な理由が
  コンテキストの要件に紐づいて説明されている。

不適切な例:

  ```md
  PostgreSQL が一番良いと思ったから。
  ```

  なぜ良いと判断したかが書かれていない。
  「検討した選択肢」の情報を踏まえた説明になっていない。
-->

## 本決定によって生じる影響と後続の作業

<!--
本セクションには、この決定によって生じる影響を記述する。良い影響だけでなく、
受け入れたトレードオフ（負の影響）も含めること。

また、この決定に伴って発生する後続の作業（Issue、タスク、設定変更等）があれば
具体的に記載する。

以下の例を参考にすること。

適切な例:

  ```md
  - マイグレーションツールとして node-pg-migrate を導入する
  - JSONB カラムには GIN インデックスを設計する必要がある（#234 で対応）
  - MySQL 経験者向けの差分ガイドを docs/guides/ に追加する（#235 で対応）
  - PostgreSQL の運用監視は既存の Datadog で対応可能（追加設定が必要）
  - トレードオフ: チーム内に PostgreSQL の深い運用知識を持つメンバーが
    おらず、初期は障害対応に時間がかかる可能性がある
  ```

  後続タスクが Issue 番号付きで列挙されており、対応漏れを防げる。
  受け入れたトレードオフも明記されている。

不適切な例:

  ```md
  PostgreSQL を使い始める。
  ```

  決定の繰り返しであり、結果として何が起きるかが記述されていない。
-->
