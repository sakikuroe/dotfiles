## 本設計がいまどの段階にあるか

<!--

- Implemented: 実装完了。本文を凍結した。

## 本設計が必要になった背景と課題

<!--
この設計が必要になった背景・課題・制約を、価値判断を交えず事実として書く。結論はここには含めず、読み手がこの節だけで「なぜこの設計が必要なのか」を理解できる粒度にする。関連する Issue があればリンクする。

例:

  ```md
  アップロード API へのレート制限導入 (#100) で過負荷は解消したが、
  検索 API・共有 API でも同様の過負荷が月 2〜3 回発生している。
  API ごとに個別対応すると実装と設定が分散するため、
  全公開 API に一貫した方式で制限を適用する設計が必要である。
  ```
-->

## 目標と非目標

<!--
この設計で達成すること (目標) と、意図的に扱わないこと (非目標) を分けて書く。非目標を明示するのは、レビューの論点が際限なく広がるのを防ぎ、スコープ外の指摘を切り分けられるようにするためである。

例:

  ```md
  - 目標: すべての公開 API に、ユーザー単位のレート制限を一貫した方式で適用する。
  - 非目標: 社内向け管理 API は対象にしない。課金プランごとの上限の差別化は次のフェーズで扱う。
  ```
-->

## 設計案

<!--
採用しようとしている設計の全体像を書く。構成図、データフロー、主要コンポーネントの責務、インターフェースなど、レビュアーが妥当性を判断できる深さで書く。細部が多い場合は小節に分けてよい。ただし、確定後の正確な仕様の正本は docs/reference であり、ここに書くのはあくまで提案時点の設計である。
-->

## 検討した代替案

<!--
検討して採用しなかった案を、長所・短所・退けた理由とともに書く。「何もしない (現状維持を続ける)」を必ず含める。現状維持と比較して初めて、変更するだけの価値があるという説得力が生まれるためである。ここを書き残しておくと、後から同じ案が再浮上したときに同じ議論を繰り返さずに済む。
-->

## 影響範囲と移行計画

<!--
この設計によって影響を受ける範囲 (システム・チーム・利用者) と、現状から新しい形へ移行する手順・段階を書く。段階的に適用する場合はその順序と判断基準を書き、問題が起きたときの切り戻しの条件と手順もあわせて書く。

例:

  ```md
  第 1 段階でアップロード API の既存実装を共通ミドルウェアへ移し、挙動が変わらないことを確認する。
  第 2 段階で検索 API・共有 API へ順に適用する。各段階で 429 の発生率を監視し、
  正常リクエストの誤制限が 0.1% を超えた場合は該当 API の制限を無効化して切り戻す。
  ```
-->

<!--
(Implemented にするまで空欄でよい)
