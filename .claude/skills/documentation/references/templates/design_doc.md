## 本設計がいまどの段階にあるか

<!--
`references/design_doc.md` で定義されている以下のいずれかのステータスを、日付とともに記載する。

- Draft: 執筆中。
- In Review: ドラフト PR としてレビュー中。
- Approved: 設計の合意が取れ、実装に進んでよい状態。同じブランチ上で実装を行う。
- Implemented: 実装完了。この時点で本文は凍結される。
- Abandoned: 設計を取り下げた。

例: Approved (2026-07-01)
例: Implemented (2026-07-15)

## 本設計が必要になった背景と課題

<!--
この設計が必要になった背景、抱えている課題、および制約事項を、執筆者の主観を交えずに事実として記述する。結論はここに含めず、読み手がこの節を読むだけで「なぜこの設計が必要なのか」を明確に理解できる粒度にとどめる。関連する Issue がある場合はリンクを記載する。

例:

  ```md
  アップロード API へのレート制限導入 (#100) で過負荷は解消したが、
  検索 API と共有 API でも同様の過負荷が月に 2〜3 回発生している。
  API ごとに個別対応すると実装や設定が分散してしまうため、
  すべての公開 API に対して一貫した方式で制限を適用する設計が必要である。
  ```
-->

## 目標と非目標

<!--
本設計で達成すべきこと (目標) と、意図的にスコープから外すこと (非目標) を明確に分けて記述する。非目標を明示することで、レビュー時の議論が際限なく発散するのを防ぎ、スコープ外の指摘を適切に切り分けられるようにする。

例:

  ```md
  - 目標: すべての公開 API に対して、ユーザー単位のレート制限を一貫した方式で適用する。
  - 非目標: 社内向け管理 API は対象外とする。課金プランごとの上限の差別化については、次のフェーズで対応する。
  ```
-->

## 設計案

<!--
採用予定の設計の全体像を記述する。構成図、データフロー、主要コンポーネントの責務、インターフェースなど、レビュアーが設計の妥当性を判断するのに十分な深さを持たせる。内容が多岐にわたる場合は、適宜小節に分割してよい。ただし、確定後の正式な仕様の正本は docs/reference であり、ここにはあくまで提案段階での設計を記載する。
-->

## 検討した代替案

<!--
検討したものの採用を見送った代替案について、それぞれの長所、短所、および不採用とした理由を記述する。その際、必ず「何もしない (現状維持)」という選択肢を含めること。現状維持と比較することで初めて、本変更を実施する価値や説得力を示すことができる。検討内容をここに書き残しておくことで、後日同様の案が再浮上した際に同じ議論を繰り返さずに済む。
-->

## 影響範囲と移行計画

<!--
本設計によって影響を受ける範囲 (システム・関係チーム・エンドユーザーなど) と、現状から新しいアーキテクチャへ移行するための手順を記述する。段階的に適用していく場合は、適用する順序とその判断基準を明記し、問題が発生した際の切り戻し条件と手順も併せて記載する。

例:

  ```md
  第 1 段階として、アップロード API の既存実装を共通ミドルウェアへ移行し、挙動が変わらないことを確認する。
  第 2 段階で検索 API と共有 API へ順次適用する。各段階で 429 エラーの発生率を監視し、
  正常なリクエストが誤って制限される割合が 0.1% を超えた場合は、該当 API の制限を無効化して切り戻しを行う。
  ```
-->

## 実装中に設計から変更した点

<!--
(ステータスが Implemented になるまでは空欄でよい)
実装の過程で、当初の設計の骨格から逸脱するような変更が生じた場合、ここで 1〜2 行程度で説明し、該当する PR へのリンクを残す。細かな変更点については PR の履歴から確認できるため、記載は不要。設計の骨格そのものが変わるほどの大規模な変更が生じた場合は、ここへの追記にとどめず、新たな Design Doc を作成することを検討する。

例:

  ```md
  - ミドルウェア層ではなく、ハンドラーの直前に配置を変更した。
    ユーザーの特定に認証情報が必要であったため (#150)。
  ```
-->
