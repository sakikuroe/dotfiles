# Design Doc の運用

Design Doc は、1 ページの ADR に収まらない大きな変更について、決定に至るまでの検討の全体を 1 か所にまとめ、レビューの土台にする文書である。`docs/` の下に置くが、性質は変更系であり、方針が確定したら凍結して以後は書き換えない。

## いつ書くか

検討の全体が 1 ページの ADR に収まらない大きな設計をするときに書く。逆に、1 つの決定として 1 ページに収まる判断は、Design Doc にせず最初から ADR ([adr.md](./adr.md)) に書く。

## 置き場と命名

`docs/design/` に、1 つの設計につき 1 ファイルを置く。ファイル名は `NNNN-<ハイフン区切りの短い題>.md` とし、`NNNN` は ADR とは独立した連番で `0001` から始める。

## 形式

次の節で構成する。本文テンプレートは [templates/design_doc.md](./templates/design_doc.md) にある。

- タイトル: `DD-NNNN: <タイトル>` の形式で、何を設計するのかがタイトルだけで分かる句で書く。
- 本設計がいまどの段階にあるか: 後述の段階の値を書く。
- 本設計が必要になった背景と課題: 設計を迫っている背景・課題・制約を、結論を含めず事実として書く。
- 目標と非目標: 達成すること (目標) と意図的に扱わないこと (非目標) を分けて書く。非目標を明示することで、レビューの論点が際限なく広がるのを防ぐ。
- 設計案: 採用しようとしている設計の全体像を、レビュアーが妥当性を判断できる深さで書く。
- 検討した代替案: 採用しなかった案を、長所・短所・退けた理由とともに書き、「何もしない (現状維持)」を必ず含める。
- 影響範囲と移行計画: 影響を受ける範囲と、段階的な適用の順序、切り戻しの条件と手順を書く。
- 本設計から ADR へ切り出した決定: 確定にあたって ADR へ切り出した決定を列挙し、相互にリンクする。
- 関連する Issue・PR・ドキュメント (任意): 関連する Issue・PR・外部資料へリンクする。

## 段階の値と遷移

段階は Draft から In Review を経て Final へ進む。それぞれの値の意味は次のとおりである。

- Draft: 執筆中であり、まだレビューに出していない。
- In Review: レビュー中である。
- Final: 方針が確定し、本文を凍結した。

## 方針が確定したら凍結する

Design Doc は 1 つの大きな変更の検討をレビューするための文書であるため、方針が確定して Final にしたら凍結し、以後は書き換えない。Final にする時点で、将来を縛る決定を ADR ([adr.md](./adr.md)) として切り出して相互にリンクし、確定後の仕様は reference に書く。凍結した後に実装や仕様が Design Doc と食い違っても、Design Doc は更新しない。現在の状態の正本はあくまで reference と ADR であり、Design Doc は「当時どう検討したか」の記録であるためである。
