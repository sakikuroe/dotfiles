# Design Doc の運用

Design Doc は、1 ページの ADR に収まらない大きな変更について、決定に至るまでの検討の全体を 1 か所にまとめ、レビューの土台にする文書である。`docs/design/` の下に置くが、性質は変更系であり、方針が確定したら凍結して以後は書き換えない。

## いつ書くか

検討の全体が 1 ページの ADR に収まらない大きな設計をするときに書く。逆に、1 つの決定として 1 ページに収まる判断は、Design Doc にせず最初から ADR ([adr.md](./adr.md)) に書く。

## 置き場と命名

`docs/design/` に、1 つの設計につき 1 ファイルを置く。ファイル名は `YYYYMMDD-<ハイフン区切りの短い題>.md` とし、`YYYYMMDD` は作成日にする。ADR と同じく、連番ではなく日付を使うのは、複数の Design Doc を並行して作成したときに番号の割り当てが git 上で衝突するのを避けるためである。

## 形式

次の節で構成する。本文テンプレートは [templates/design_doc.md](./templates/design_doc.md) にある。

- タイトル: `DD-YYYYMMDD: <タイトル>` の形式で、何を設計するのかがタイトルだけで分かる句で書く。
- 本設計がいまどの段階にあるか: 後述の段階の値を書く。
- 本設計が必要になった背景と課題: 設計を迫っている背景・課題・制約を、結論を含めず事実として書く。
- 目標と非目標: 達成すること (目標) と意図的に扱わないこと (非目標) を分けて書く。非目標を明示することで、レビューの論点が際限なく広がるのを防ぐ。
- 設計案: 採用しようとしている設計の全体像を、レビュアーが妥当性を判断できる深さで書く。
- 検討した代替案: 採用しなかった案を、長所・短所・退けた理由とともに書き、「何もしない (現状維持)」を必ず含める。
- 影響範囲と移行計画: 影響を受ける範囲と、段階的な適用の順序、切り戻しの条件と手順を書く。
- 実装中に設計から変更した点: 実装中に骨格が変わった場合に、1〜2 行と PR リンクで追記する欄。
- 本設計から ADR へ切り出した決定: 確定にあたって ADR へ切り出した決定を列挙し、相互にリンクする。
- 関連する Issue・PR・ドキュメント (任意): 関連する Issue・PR・外部資料へリンクする。

## 段階の値と遷移

段階は Draft から始まり、Approved を経て Implemented で凍結するのが基本の流れである。それぞれの値の意味は次のとおり。

- Draft: 執筆中であり、まだレビューに出していない。
- In Review: レビュー中である。ドラフト PR として提出し、PR のコメントで議論する。
- Approved: 設計が合意に達し、この設計で実装に進んでよい状態である。ドラフト PR 上でステータスを Approved に変更して push する。この時点では凍結しない。同じブランチで引き続き実装を行うためである。
- Implemented: 実装が完了し、本文を凍結した。PR のマージ時にこの値にする。凍結後は本文を書き換えず、現在の状態の正本は reference と ADR とする。
- Abandoned: 設計を取りやめた。実装に進まず中止した場合にこの値にする。

Approved と Implemented を分けているのは、設計レビューの完了と実装の完了が異なる時点であるためである。Approved は「この方向で作ってよい」という合意であり、Implemented は「作り終えて凍結した」という事実である。

## 方針が確定したら凍結する

Design Doc は 1 つの大きな変更の検討をレビューするための文書であるため、Implemented にしたら凍結し、以後は本文を書き換えない。Approved にする時点で、将来を縛る決定を ADR ([adr.md](./adr.md)) として切り出して相互にリンクし、確定後の仕様は reference に書く。凍結した後に実装や仕様が Design Doc と食い違っても、Design Doc は更新しない。現在の状態の正本はあくまで reference と ADR であり、Design Doc は「当時どう検討したか」の記録であるためである。

ただし、実装中に設計の骨格が変わった場合に限り、テンプレートの「実装中に設計から変更した点」の欄に 1〜2 行と PR へのリンクで追記する。細かい差分は PR の履歴に残るので書かなくてよい。骨格が変わるほど大きくずれた場合は、追記ではなく新しい Design Doc を起こすことを検討する。
