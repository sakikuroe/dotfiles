# Design Doc の運用

Design Doc は、1 ページの ADR に収まらない大規模な変更について、決定に至るまでの検討プロセス全体を 1 か所にまとめ、レビューの土台とする文書である。`docs/design/` 配下に配置するが、あくまで変更の設計プロセスを管理する文書であるため、方針の確定後は凍結し、それ以降は更新しない。

## 作成のタイミング

検討の全体像が 1 ページの ADR に収まらないような、大規模な設計を行う際に作成する。反対に、単一の決定事項として 1 ページに収まる場合は、Design Doc を作成せず最初から ADR ([adr.md](./adr.md)) に記載する。

## 配置場所と命名規則

`docs/design/` 配下に、1 つの設計につき 1 ファイルを配置する。ファイル名は `YYYYMMDD-<ハイフン区切りの簡潔なタイトル>.md` とし、`YYYYMMDD` には作成日を指定する。ADR と同様に連番ではなく日付を使用するのは、複数の Design Doc を並行して作成する際に、git 上で番号の割り当てが衝突するのを防ぐためである。

## 文書の構成

文書は以下の項目で構成する。本文のテンプレートは [templates/design_doc.md](./templates/design_doc.md) に用意されている。

- タイトル: `DD-YYYYMMDD: <タイトル>` の形式とし、何を設計するのかがタイトルだけでわかるように記述する。
- 現在のステータス: 後述するステータスの値を記載する。
- 背景と課題: 今回の設計が必要になった背景や直面している課題、制約条件を、結論を含めずに客観的な事実として記載する。
- 目標と非目標: 達成すべきこと (目標) と、意図的に扱わないこと (非目標) を分けて記載する。非目標を明示することで、レビューの論点が際限なく広がるのを防ぐ。
- 設計案: 採用予定の設計の全体像を、レビュアーが妥当性を判断できる十分な深さで記載する。
- 検討した代替案: 不採用となった案について、長所・短所・不採用の理由とともに記載する。ここには「何もしない (現状維持)」という選択肢も必ず含める。
- 影響範囲と移行計画: 影響を受ける範囲、段階的な適用の順序、および切り戻しの条件と手順を記載する。
- 実装時の変更点: 実装中に設計の骨格が変わった場合に、概要を 1〜2 行で書き添え、PR へのリンクを追記する欄。
- ADR への切り出し: 方針確定にあたり、本設計から ADR へ切り出した決定事項を列挙し、相互にリンクを張る。
- 関連する Issue・PR・ドキュメント (任意): 関連する Issue や PR、外部資料などへのリンクを記載する。

## ステータスの値と遷移

ステータスは Draft から始まり、Approved を経て Implemented で凍結するのが基本の流れである。それぞれの値の意味は以下のとおり。

- Draft: 執筆中の状態。まだレビューには出していない。
- In Review: レビュー中の状態。ドラフト PR として提出し、PR のコメント上で議論を行う。
- Approved: 設計の合意が形成され、実装に進んでよい状態。ドラフト PR 上でステータスを Approved に変更して push する。ただし、同じブランチで引き続き実装を行うため、この時点ではまだ文書を凍結しない。
- Implemented: 実装が完了し、文書を凍結した状態。PR をマージするタイミングでこの値に変更する。凍結後は本文を更新せず、現状の仕様の正本は reference と ADR に委ねる。
- Abandoned: 設計を取り下げた状態。実装に進まずに中止した場合にこの値に設定する。

Approved と Implemented を分けているのは、設計レビューの完了と実装の完了がそれぞれ異なるタイミングであるためである。Approved は「この方針で実装を進めてよい」という合意を示すのに対し、Implemented は「実装が完了し、文書を凍結した」という事実を示す。

## 方針確定後の凍結

Design Doc は大規模な変更を検討・レビューするための文書であるため、ステータスを Implemented にした時点で凍結し、以後は本文を更新しない。ステータスを Approved にする段階で、将来の制約となる決定事項を ADR ([adr.md](./adr.md)) に切り出して相互にリンクし、確定後の仕様は reference に記載する。凍結後に実装や仕様が Design Doc の内容と食い違ったとしても、Design Doc は更新しない。現在の仕様の正本はあくまで reference と ADR であり、Design Doc は「当時どのように検討したか」を残す記録として位置づけられるためである。

ただし、実装中に設計の骨格が変わった場合に限り、テンプレートの「実装時の変更点」の欄に概要を 1〜2 行書き添え、PR へのリンクを追記する。細かい差分は PR の履歴として残るため記載する必要はない。設計の骨格が変わるほどの大きな変更が生じた場合は、追記で済ませるのではなく、新たな Design Doc の作成を検討する。
