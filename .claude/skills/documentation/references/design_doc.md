# Design Doc の運用

Design Doc は、1 ページの ADR に収まらない大きな変更について、決定に至るまでの検討の全体を 1 か所にまとめ、レビューの土台にする文書である。`docs/` の下に置くが、性質は変更系であり、レビューに合意して Approved になったら凍結して以後は書き換えない。

## いつ書くか

判断の基準は「間違えたときの手戻りの大きさ」である。具体的には、実装に数日以上かかる、作り方が複数考えられる、他のメンバーや他システムに影響する、一度作ると戻しにくい、のうち複数に該当する変更は、着手前に Design Doc を書く。基準に迷う場合は、30 分程度で書ける 1 ページを書く側に倒す。書くべき変更で書かれない事態のほうが、不要な Design Doc が 1 つ増えるより高くつくためである。

逆に、これらにほとんど該当せず、1 つの決定として 1 ページに収まる判断は、Design Doc にせず最初から ADR ([adr.md](./adr.md)) に書く。

## 置き場と命名

`docs/design/` に、1 つの設計につき 1 ファイルを置く。ファイル名は `YYYYMMDD-<ハイフン区切りの短い題>.md` とし、`YYYYMMDD` は作成日にする。ADR と同じく、連番ではなく日付を使うのは、複数の Design Doc を並行して作成したときに番号の割り当てが git 上で衝突するのを避けるためである。

## 形式

次の節で構成する。本文テンプレートは [templates/design_doc.md](./templates/design_doc.md) にある。

- タイトル: `DD-YYYYMMDD: <タイトル>` の形式で、何を設計するのかがタイトルだけで分かる句で書く。
- 本設計がいまどの段階にあるか: 後述の段階の値を書く。
- 本設計が必要になった背景と課題: 設計を迫っている背景・課題・制約を、結論を含めず事実として書く。
- 目標と非目標: 達成すること (目標) と意図的に扱わないこと (非目標) を分けて書く。非目標を明示することで、レビューの論点が際限なく広がるのを防ぐ。
- 設計案: 採用しようとしている設計の全体像を、レビュアーが妥当性を判断できる深さで書く。
- 検討した代替案: 採用しなかった案を、長所・短所・退けた理由とともに書き、「何もしない (現状維持)」を必ず含める。
- 影響範囲と移行計画: 影響を受ける範囲と、段階的な適用の順序、切り戻しの条件と手順を書く。
- 本設計から ADR へ切り出した決定: 確定にあたって ADR へ切り出した決定を列挙し、相互にリンクする。
- 関連する Issue・PR・ドキュメント (任意): 関連する Issue・PR・外部資料へリンクする。

## 段階の値と遷移

段階は Draft → In Review → Approved の順に進み、そこから Implemented または Abandoned のいずれかに至る。それぞれの値の意味と遷移のタイミングは次のとおりである。

- Draft: 執筆中であり、まだレビューに出していない。設計の全体像が書けたら In Review へ進める。
- In Review: ドラフト PR 上でレビューを受けている。
- Approved: 設計のレビューに合意できた段階である。合意できた時点でこの値に変更して push し、以後は同じブランチで実装に進む。この段階に達した時点で本文を凍結し、以後は書き換えない。
- Implemented: 実装・ADR・ドキュメント更新までを含めた PR 全体のレビューが Approve され、マージする直前の最終 push でこの値に変更する。マージした後に書き換えるという意味ではない。
- Abandoned: この変更を実装せずに取りやめた段階である。Approved の後に取りやめた場合だけでなく、Draft や In Review の段階で取りやめた場合もこの値に変更する。

## Approved 以降は本文を凍結する

Design Doc は、設計のレビューに合意した Approved の時点から本文を凍結し、以後は書き換えない。凍結の開始点は実装完了 (Implemented) ではなく合意 (Approved) であり、Approved から Implemented または Abandoned に至るまでの実装期間中も、本文は凍結されたままである。Draft や In Review の段階で Abandoned にする場合も、それ以降は書き換えず、変更を取りやめた時点の内容をそのまま残す。いずれの場合も、誤字の修正やリンク切れの補修は例外として許容するが、設計案や検討した代替案などの内容には手を入れない。

凍結中の唯一の例外は、「本設計から ADR へ切り出した決定」の一覧への追記である。Approved 以降の実装期間中に、あとから「なぜ」と聞かれそうな寿命の長い決定をした場合、その決定は ADR として随時 Proposed で切り出してよい (documentation スキルの [adr.md](./adr.md) を参照)。この場合、切り出した ADR へのリンクをこの一覧に追記することは、凍結中であっても許容する。ADR 自身が Accepted になった後もステータス行だけは書き換え可能なのと同じく、この追記は決定の内容そのものを書き換えるものではなく、Design Doc から見て現在有効な決定がどこにあるかを指し示すための更新だからである。

Approved にする時点で、その時点までに固まっている将来を縛る決定は ADR として切り出して相互にリンクし、確定後の仕様は reference に書く。凍結後に実装や仕様が Design Doc と食い違っても、Design Doc は更新しない。現在の状態の正本はあくまで reference と ADR であり、Design Doc は「当時どう検討したか」の記録であるためである。
