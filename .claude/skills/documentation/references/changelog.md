# CHANGELOG の運用

CHANGELOG は、バージョン間の変更を利用者の言葉で記録する恒常系の文書である。書き方は Keep a Changelog (https://keepachangelog.com/)、バージョン番号は Semantic Versioning (https://semver.org/) に従う。雛形は [templates/changelog.md](./templates/changelog.md) にある。

## 何を書き、何を書かないか

利用者にとって意味のある変更を、利用者の言葉で書く。「何が追加され、何が変わり、何が直ったか」を、その変更を受け取る人の視点でまとめる。

内部実装の詳細は書かない (→ PR / コミット)。コミットログをそのまま貼り付けない。コミットは開発者が読む記録、CHANGELOG は利用者が読む記録であり、目的が違う。コミットから機械的に生成した場合でも、利用者向けに書き直す。

## 構成

- ファイル先頭に `[Unreleased]` 節を置き、未リリースの変更をここに貯める。リリース時に、この内容をバージョン節へ移す。
- 各バージョンは `## [x.y.z] - YYYY-MM-DD` の見出しで、新しいものを上に書く。
- 各節の中では、次の分類のうち該当するものだけを使う。

| 分類 | 用途 |
| --- | --- |
| Added | 新機能 |
| Changed | 既存機能の変更 |
| Deprecated | 近く削除予定の機能 |
| Removed | 削除した機能 |
| Fixed | 不具合の修正 |
| Security | 脆弱性に関する変更 |

## Semantic Versioning との対応

バージョン番号 `x.y.z` は変更の性質を表す。

- x (major): 後方互換性を壊す変更。
- y (minor): 後方互換を保つ機能追加。
- z (patch): 後方互換を保つ不具合修正。

CHANGELOG の分類とバージョンの上げ方は対応する。後方互換を壊す Changed や Removed があれば major、Added があれば minor、Fixed だけなら patch を上げる。

## コミットとの関係

コミットメッセージを規約に沿って書いておくと、リリース時に変更を拾い上げやすくなる。ただし CHANGELOG への記載は手作業で行い、利用者にとっての意味が伝わる粒度と言葉に整える。コミット 1 件と CHANGELOG 1 行は必ずしも一対一ではなく、複数のコミットを 1 つの変更点としてまとめてよい。
