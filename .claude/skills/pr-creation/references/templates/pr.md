## 本 PR で行った変更の概要

<!--
本 PR で何をどのように変更したかを記述する。

レビュアーが差分を確認する前に変更の全体像を把握できるよう記述すること。変更が複数のファイルやモジュールにまたがる場合は、どのような構造の変更なのかが文章の流れから伝わるように構成する。

なお、Issue の背景をここで繰り返す必要はない。

以下の例を参考にすること。

適切な例:

  ```md
  CI パイプラインのテストジョブを並列化した。約 800 件のテスト
  ケースをモジュール単位で 4 つのグループに分割し、GitHub Actions
  の matrix strategy を使って同時実行する構成に変更した。

  主な変更箇所は以下のとおりである。
  - `.github/workflows/ci.yml` に matrix 定義と結果集約ジョブを追加
  - `scripts/split-tests.sh` を新規作成（テストファイルをグループに
    振り分けるスクリプト）
  - `pytest.ini` に並列実行時のグループ指定オプションを追加
  ```

  冒頭で変更の要点（テストジョブの並列化）を述べた上で、具体的な変更箇所を構造的に示している。これにより、レビュアーは差分を開く前に、どのファイルにどのような変更があるかを把握できる。

不適切な例:

  ```md
  CI を改善した。
  ```

  何をどう変えたのかが分からず、差分を読むまで変更の内容を把握できない。
-->

## 実装方針の判断理由

<!--
（任意）
Design Doc がある変更の場合、実装方針はそちらで合意済みであるため、ここで内容を繰り返す必要はない。Design Doc へのリンクを記載し、実装がその内容から乖離した点とその理由のみを記述する。Design Doc どおりに実装した場合は、リンクのみでよい。
Design Doc がない変更の場合は、採用した方法に加え、検討したが見送った方法とその理由をここに直接記述する。
将来の実装を縛るような判断を新たに行った場合は、その決定の正本は PR 本文ではなく ADR に置き、ここでは ADR へのリンクを記載する（documentation スキルの references/adr.md を参照）。
判断理由を記述する際は、採用した方法と見送った方法の双方、そして見送った理由を明記すること。この記録は、将来他の開発者が「なぜこの実装にしたのか」と疑問を持った際の回答となる。

変更が軽微で意図が自明な場合（typo 修正、フォーマット変更等）は省略して構わない。

以下の例を参考にすること。

適切な例:

  ```md
  テストの分割方法として、モジュール単位での静的な分割を採用した。
  代替案として実行時間に基づく動的分割（pytest-split 等）も検討
  したが、過去の実行時間データの収集基盤が現時点では存在しないため、
  まずは導入コストの低い静的分割で開始し、効果を計測した上で動的
  分割への移行を検討する方針とした。
  ```

  採用した方法（静的分割）と見送った方法（動的分割）の双方が述べられ、見送った理由（収集基盤がないこと）も具体的に示されている。この記録を残すことで、将来再検討する際に同じ議論を繰り返さずに済む。

不適切な例:

  ```md
  pytest-split は使わないことにした。
  ```

  なぜ使わないのかが述べられていないため、将来同じ選択肢が再浮上した際に、当時の判断根拠を辿ることができない。
-->

## 本 PR では取り扱わないこと

<!--
（任意）
意図的に本 PR の対象外とした内容を記述する。これを明記することで、レビュー時の「対応が漏れているのではないか」という指摘を未然に防ぐことができる。後続の対応が必要な場合は、別の Issue を作成してリンクすること。

以下の例を参考にすること。

適切な例:

  ```md
  テストケース自体の削減や統廃合は本 PR では対応しない。実行時間の
  短縮はあくまで並列化によって実現する。テスト内容の見直しは #790
  にて別途検討する。
  ```

  何を対象外としたか（テストの削減）と、その後続対応（#790）が明示されている。

不適切な例:

  ```md
  関係ないことはやっていない。
  ```

  何を「関係ない」と判断したのかが示されておらず、範囲の限定として機能していない。
-->

## 動作を確認する手順

<!--
（任意）
本 PR の変更が意図どおりに動作することを確認する手順を記述する。レビュアーがローカル環境や検証環境で再現する際の手順として使えるよう、具体的に記載すること。

以下の例を参考にすること。

適切な例:

  ```md
  1. 本ブランチをチェックアウトする
  2. feature/parallel-ci ブランチに push する
  3. GitHub Actions の「CI」ワークフローが 4 つの並列ジョブで実行されることを確認する
  4. 全ジョブ完了後、PR のステータスチェックに「CI Results」が 1 つだけ表示されることを確認する
  5. いずれかのテストを意図的に失敗させて push し、PR のステータスが失敗になることを確認する
  ```

  手順が番号付きで記述されており、正常系（手順 3〜4）と異常系（手順 5）の両方が含まれている。

不適切な例:

  ```md
  動作確認済み。
  ```

  何をどのように確認したのかがわからないため、レビュアーが再現確認を行えない。
-->
