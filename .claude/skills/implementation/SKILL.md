## 概要

作業ブランチで実装を進め、検証を通し、ローカルコミットを積む段階である。対応する Issue がある場合はその内容を実装するが、Issue を前提とはせず、ユーザーの指示から直接始めることもできる。

この段階を通じて守るべき姿勢がひとつある。計画にない変更を行いたくなった場合や、仕様・実装方針で分からないことが出てきた場合は、推測で進めずに作業を止め、ユーザーに報告・相談する。実装は差分が積み上がる段階であり、誤った前提のまま進むほど手戻りが大きくなるためである。

## 手順

### 事前確認

まず、何を実装するのかの拠り所を確認する。対応する Issue がある場合はそれを特定し、ない場合はユーザーの指示内容を確認する。あわせて作業ブランチを確認し、まだ存在しない場合や現在のブランチが作業用として適切でない場合は、どのブランチで作業するかをユーザーに確認してから用意する。作業ツリーに意図しない差分が残っている場合は、勝手に処理せず、扱いをユーザーに相談する。

次に、実装計画の有無をユーザーに確認する。計画がある場合は所在（Issue のコメント、計画文書など）を教えてもらい、その内容を読んで実装範囲を把握する。計画がない場合は、Issue や指示内容をもとに自分で計画を提案し、ユーザーの承認を得てから進める。軽微な作業などで計画の確認自体が不要だとユーザーが判断した場合は、このステップを省略してよい。

最後に、検証の方法を確認する。テスト・lint・build などの実行コマンドはリポジトリーごとに異なるため、着手前にユーザーに聞いておく。実装が終わってから検証手段を探し始めると、その時点で手が止まってしまうためである。

### 進捗の記録

事前確認が済んだら、実装に着手する直前に、対応 Issue へ進捗コメントを投稿する。コメント本文をファイルに書き出し、`bash .claude/skills/implementation/scripts/add_progress_comment.sh <issue番号> <body_file>` で「状態: 実装中」を記録する。対応する Issue がない場合、このステップはスキップする。

### 実装とコミット

計画に沿って実装を進める。作業中は、論理的な区切りがつくたびに適切なタイミングでコミットを積む。細かく積んでおくことで、問題が起きたときに戻れる地点が増え、レビューでも変更の意図を追いやすくなるためである。

コミットは自分で `git commit` を実行せず、git-committer サブエージェントに委譲する。委譲の際は、作業ディレクトリーと、コミットする単位・意図を伝える。

実装中に計画にない変更が必要だと分かった場合は、その変更を行う前にユーザーに相談する。仕様や実装方針で判断に迷うことが出てきた場合も同様で、推測で進めずに作業を止めてユーザーに聞く。

### 検証

実装が一区切りついたら、事前確認で聞いておいたコマンドで検証を実行する。失敗した場合は原因を調べて修正し、再度実行する。自力で解消できない失敗に当たった場合は、状況を整理してユーザーに相談する。

検証で修正が生じた場合は、その修正もコミットとして積む。

## この段階の完了条件

- [ ] 実装が計画（または承認された提案）の範囲を満たしている。
- [ ] 検証が通っている。
- [ ] 変更がすべてコミット済みである。
- [ ] 対応 Issue がある場合、進捗コメントが投稿済みである。

## スクリプト

- [add_progress_comment.sh](./scripts/add_progress_comment.sh): `bash .claude/skills/implementation/scripts/add_progress_comment.sh <issue番号> <body_file>` で Issue に進捗コメントを投稿する。実装開始時に `状態: 実装中` を記録するために使う。

## 連携サブエージェント

- git-committer: コミットを担う。作業ディレクトリとコミットの単位・意図を伝えて委譲する。
