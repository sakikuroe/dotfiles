# 日本語技術文書作成のための記述ルール

本ドキュメントは、日本語技術文書を作成する際の記述ルールを定めたものである。文書全体の品質と一貫性を確保するために、本ルールに従って記述すること。

## 文末表現・箇条書き・スペース

### 文末表現

体言止めで文を終える場合には句点を付けないこと。一方、「である」のように述語で終わる文には句点を付ける。例えば `ファイルの保存先を指定する。` は述語で終わるため句点を付けるが、`保存先のパス` のような体言止めには句点を付けない。`ファイルの保存先を指定。` のように、述語から句点だけを取り除いた中途半端な形は避けること。

### 箇条書き

箇条書きやリストの各項目も、1 文として書く場合は文末に句点を付けること。体言止めや単語のみで構成される項目には句点を付けず、同一のリスト内では文末表現 (句点の有無、体言止めか文末表現か) を統一すること。

正しい記述例:

- 本機能は、任意のファイルを指定できます。
- 次の手順でインストールを行います。
- 環境変数の設定
- 依存パッケージの導入

### スペース

日本語と英単語、または半角数字の間には必ず半角スペースを挿入すること。例えば `Python 3.12 をインストールする。` が正しく、`Python3.12をインストールする。` は誤りである。

## 括弧、記号、書式

### 括弧

括弧は半角の `(` `)` (U+0028、U+0029) を使うこと。全角括弧 `（` `）` (U+FF08、U+FF09) は使用しない。例えば `設定値 (デフォルトは 0) を変更する。` が正しく、`設定値（デフォルトは 0）を変更する。` は誤りである。

左括弧 `(` の前と右括弧 `)` の後ろには原則として半角スペースを挿入する。ただし、文頭や数式、プログラムコードの場合はスペースを挿入しない。例えば、`f(x) = y` のような数式や、`print("hello")` のようなコード中では括弧の前後にスペースを入れない。

### 引用符

引用符は半角のダブルクォーテーション `"` (U+0022) を用いること。全角のダブルクォーテーション `"` `"` (U+201C、U+201D) は使用しない。また、引用符を強調の目的で使用しないこと。例えば `この概念は "Ownership" と呼ばれている。` のように用語の導入に用いるのは正しいが、`この機能は "非常に" 重要である。` のような強調方法は禁止する。

### コード、コマンドの表記

ファイル名、コマンド名、関数名、変数名などをインラインで記載する場合は、バッククォートで囲むこと。複数行のコードやコマンドを記載する場合は、コードブロックを使用し、言語名を指定する。例えば  "`git commit` を実行する。" が正しく、バッククォートなしで "git commit を実行する。"と書くのは誤りである。

## 表現

### 主語の明示

動作の主体が曖昧になる場合は、主語を明示すること。特に手順書では、操作の実行者 (ユーザー、システム、管理者など) を明確にする。例えば `ユーザーがフォームを送信すると、システムが確認メールを送信する。` が正しく、`フォームを送信すると、確認メールが送信される。` では誰が何をするのかがわからない。

### 能動態と受動態

文章のわかりやすさのため、原則として能動態を使用すること。ただし、動作の対象を強調する場合や、動作主体を特定する必要がない場合には、受動態を使用してよい。例えば `システムがエラーログを記録する。` が基本だが、`エラーログはシステム起動時に自動で作成される。` のように動作主体を問わない場面では受動態でよい。

### 二重否定の回避

二重否定は使用せず、肯定表現で書き換えること。例えば `この提案が不適切でないとは言えない。` のような二重否定は避け、`この提案は適切かどうか疑わしい。` のように書く。

### 曖昧な表現の回避

"など"、"等"、"いろいろ" といった曖昧な表現は、可能な限り具体的な内容に置き換えること。やむを得ず使用する場合は、代表的な要素を 2 つ以上列挙してから使用する。例えば `JSON、YAML、TOML などの設定ファイル形式に対応している。` は正しいが、`JSON などに対応している。` では何に対応しているのか伝わらない。

## 外国語、外来語

### 外国語の固有名詞、技術用語

製品名、固有の技術名、または日本語での定訳や定着した表記が存在しない用語については、可能な限り元の言語で記載すること。一方、"コンパイル"、"デバッグ" など、すでに日本語 (カタカナ語) として広く定着している用語についてはカタカナで記載する。迷った場合は、JIS 規格や各分野の標準的な用語集を参照すること。

例えば `このモデルは Transformer をベースにしている。` のように、固有名詞やまだ定着していない技術用語は英語で記載する。逆に `ソースコードをコンパイルする。` のように定着した用語はカタカナを用いる。`このモデルはトランスフォーマーをベースにしている。` のように固有名詞をカタカナにしたり、`ソースコードを compile する。` のように定着した用語を英語にしたりするのは誤りである。

### 外来語の長音符号

外来語の長音符号 (ー) は、原則として省略しないこと。ただし、固有名詞としての製品名やサービス名がもともと長音符号を省略した表記を採用している場合は、その表記に従う。例えば `サーバーを再起動する。` が正しく、`サーバを再起動する。` は誤りである。同様に `コンピューター` を `コンピュータ` と省略しない。

## 漢字、ひらがな、送り仮名

### 漢字とひらがなの使い分け

形式名詞、補助動詞、副助詞は原則としてひらがなで記載すること。以下に代表的な使い分けの例を示す。

| 種別 | ひらがな (正) | 漢字 (誤) |
|---|---|---|
| 形式名詞 | こと、もの、とき、ところ、わけ、ほう、うち、ため | 事、物、時、所、訳、方、内、為 |
| 補助動詞 | ～してください、～してほしい、～してみる、～しておく、～していく、～してくる | ～して下さい、～して欲しい、～して見る、～して置く、～して行く、～して来る |
| 副詞・接続詞 | さらに、すでに、および、ならびに、または、あるいは、ただし、なお | 更に、既に、及び、並びに、又は、或いは、但し、尚 |
| 副助詞等 | ～のとおり、～のほか、～のうえ、～のうち、～のもと | ～の通り、～の他、～の上、～の内、～の下 |

ただし、漢字が本来の意味で使われる場合は漢字で記載する。例えば `設定を保存しておくことを推奨する。` の「こと」は形式名詞なのでひらがなで書くが、`事の発端は設計段階にさかのぼる。` の「事」は本来の意味 (出来事) を持つため漢字で書く。

### 送り仮名

送り仮名は省略しないこと。例えば `参加の申し込みを受け付ける。` が正しく、`参加の申込みを受付ける。` のように送り仮名を省くのは誤りである。

### 同義語、表記ゆれの統一

同一の文書内で、同じ概念を指す語は統一すること。文書の冒頭や付録に用語集を設け、使用する表記を定義することを推奨する。例えば「ファイルを作成する」と「ファイルを生成する」のように同じ操作を別の動詞で書くと、読み手は異なる操作かと誤解する。

## 用語

### 半角カタカナ、全角英数字、全角の ASCII 記号

半角カタカナ、全角英数字、ASCII 記号の全角版 (例: `＋`、`＿`、`！`) はいずれも使用しないこと。例えば `Enter` が正しく、`Ｅｎｔｅｒ` は誤りである。

## 見出し、構成

### 見出しの書き方

見出しは体言止めで統一し、末尾に句点は付けないこと。見出しのレベル (階層) は、内容の包含関係に基づいて適切に設定する。例えば `## データベースの設計` が正しく、`## データベースの設計について説明する。` のように文にしたり句点を付けたりするのは誤りである。また `##` の直下に `####` を置くようなレベルの飛ばし方も避けること。

### 図表、コードブロックの参照方法

図、表、コードブロックには通し番号とキャプションを付けること。本文中で図表を参照する際は、`表 1 に対応する OS の一覧を示す。` のように番号で参照し、`下の表に対応する OS を示す。` のような曖昧な参照は避ける。図表は、本文中で参照する位置のできるだけ近くに配置すること。

## 例文

以下に、本文書で定めたルールに従った例文を示す。

```text
Docker のインストール手順を以下に示す。まず、公式リポジトリーの GPG 鍵を取得し、apt のソース一覧に追加する。その後、`apt update` でパッケージ情報を更新してから `docker-ce` を導入する。インストールが完了したら、`docker run hello-world` を実行して動作を確認すること。詳細は公式ドキュメントの "Get Docker" を参照してほしい。
```

```text
Git と Subversion は、共にソースコードのバージョン管理に用いられるツールであるが、設計思想に大きな違いがある。

Git は分散型のバージョン管理システムであり、各開発者のローカル環境にリポジトリーの完全なコピーを持つ。そのため、ネットワークに接続していない状態でもコミットやブランチの作成が可能であり、オフライン環境での作業効率が高い。また、ブランチの作成とマージが軽量に設計されているため、feature branch を用いた並行開発に適している。

一方、Subversion は集中型のバージョン管理システムであり、すべての履歴が中央のサーバーに保存される。クライアントは必要なファイルだけをチェックアウトすればよいため、巨大なバイナリーファイルを多数含むプロジェクトではディスク使用量を抑えやすい。さらに、ディレクトリー単位でのアクセス制御が容易であり、部門ごとに閲覧権限を分けたい組織での運用に向いている。
```
