# 散文構造のルール

返答、Issue、PR、ドキュメントなど、あらゆる文章は整理された散文として書くこと。箇条書きに頼るのではなく、背景・課題・解決策といった論理の展開が、段落を通じて自然に流れるように構成する。ただし、理解を助けるための表や図、数式の使用は問題ない。

このルールは、箇条書きの記号を外しただけの、短い命令文や体言止めを句点で並べただけの文章を散文と錯覚してしまうのを防ぐために設けている。そのような文章は一文ごとに意味が途切れ、文脈のつながりや理由が欠落しているため、読み手はただ情報の羅列を暗記させられるような負担を感じる。目指すのはその逆で、文と文が接続詞や論理によって結びつき、規則には理由が添えられ、作業手順は時間の流れに沿って読める文章である。

なお、適用の強さは文書の種類や長さに応じて調整してよい。数行で終わる短い返答では簡潔さを優先してよく、コミットメッセージの変更点一覧のように箇条書きが慣習として定着している箇所では、その慣習に従う。

## 基本原則

第一に、文章は単なる「列挙」ではなく「展開」を意識して書く。前の文を次の文の前提として構成し、順接・逆接・補足といった文脈のつながりを「したがって」「ただし」「一方で」「なお」「つまり」などの接続表現で明確にする。すべての文に接続詞が必要なわけではないが、読者が「なぜ今この話が出てきたのか」と迷いそうな箇所には、適切な接続表現を補うこと。

第二に、規則や指示には原則として理由や目的を添える。単に「URL は正規化する」と書くのではなく、「同じページを異なる表記で二重に記録するのを防ぐため、記録前に URL を正規化する」のように記述する。理由が明白でくどくなる場合は省略してもよいが、迷ったときは書き添える。理由が明示された規則は納得感を与え、応用も利きやすくなるからである。

第三に、一つ一つの文を省略のない完全な文として記述する。体言止めの多用 (「〜を確認。」「〜を保存。」)、主語や対象が欠落した断片的な表現、短い命令形の連続は避けること。一方で、一文に条件を詰め込みすぎて読点だらけになるのも読みにくいため、適度に長短の文を織り交ぜる。

第四に、文書全体で文体を統一する。指示書や仕様書、ノートは常体 (だ・である調) を基本とし、読者に直接語りかける文書では敬体 (です・ます調) を使用する。どちらを選ぶ場合でも、ひとつの文書内で文体を混在させないこと。

## 手順・プロセスの書き方

作業手順は、番号付きリストに頼るのではなく、時間の経過に沿った段落として記述する。「まず〜する。次に〜を決める。〜が終わったら〜し、読み終えたら〜する。最後に〜だけを返す」といった具合に、順序を示す言葉や「〜したら」「〜が済んだら」という完了の表現を使えば、番号がなくても手順は十分に伝わる。条件分岐や失敗時の対応についても、該当する手順のすぐ後で「失敗した場合は〜にフォールバックし、それでも〜なら〜して終了する」のように一続きの文章として書き、別の項目として切り離さないこと。番号付きの箇条書きを使ってよいのは、読者が上から順に消し込むチェックリストのように、各項目が独立していて番号や消し込みに意味がある場合に限る。記法の詳細は [markdown_rule.md](./markdown_rule.md) に従う。

段落を変えるタイミングは、作業の意味的な区切り (準備、取得、執筆、報告など) に合わせる。1 ステップを 1 段落に機械的に対応させる必要はなく、細かなステップはひとつの段落にまとめてよい。

## 規則・条件の書き方

複数の条件を列挙する場合は、可能な限り一文にまとめる。例えば「A、B、C のいずれかに当てはまれば失敗とみなす」と表現すれば、3 行分の箇条書きを 1 文に圧縮できる。列挙する要素が多く一文では読みにくい場合は、「具体的には、〜を取り除き、〜を落とし、〜に揃える」のように動詞でつなぎながら並べることで、単なる羅列ではなく流れのある説明として読ませることができる。

また、禁止事項を記載する際は「〜してはいけない」という禁止だけで終わらせず、その理由や代替案を必ず添えること。「言い換えた断片を書いてはならない。照合の役に立たないからである」のように、禁止と理由 (および代替の行動) をセットにして記述する。

## 箇条書きを使ってよい場合

基本は散文だが、例外もある。コードの断片、コマンド、設定例、データ形式の定義、エラーメッセージは、原文のままコードブロックで提示すること。また、表形式が本質的な対照表や、読者が上から順に確認していくチェックリストには、箇条書きを使用してよい。

使い分けの基準は、要素の間に論理的なつながりがあるかどうかである。つながりがあるなら散文、それぞれが独立して参照される項目群なら箇条書きを選ぶ。なお、箇条書きを使用する場合でも、各項目は体言止めを避け、1〜2 文の完全な文章で記述すること。

## 悪い例と良い例

悪い例。記号のない箇条書きになっている。

> URL を正規化する。フラグメントを除去。utm 系パラメータを除去。末尾スラッシュを除去。スキームとホストは小文字。frontmatter には正規化後の URL を書く。

良い例。目的で始まり、動詞で受けながら一続きに説明している。

> 同じページを別表記の URL で二重に記録してしまわないよう、URL は記録や比較の前に必ず正規化しておく。具体的には、フラグメントとトラッキング用のクエリパラメータを取り除き、末尾のスラッシュを落とし、スキームとホストを小文字に揃える。frontmatter に書くのはこの正規化後の URL である。

悪い例。手順が命令の羅列になっていて、失敗時の扱いも切り離されている。

> 1. curl で取得する。
> 2. 失敗判定: 終了コード非 0、HTTP 非 2xx、0 バイト。
> 3. 失敗時は WebFetch にフォールバック。
> 4. それも失敗したら報告して終了。

良い例。ひとつの流れとして読め、判定の意味も文の中に収まっている。

> まず curl で原文を取得する。終了コードが 0 以外だったり、HTTP ステータスが 2xx 以外だったり、取得サイズが 0 だったりした場合は失敗とみなし、WebFetch にフォールバックする。それでも取得できなければ、失敗した旨を報告して終了する。

悪い例。PR の説明が変更点の羅列になっていて、なぜ変更したのかが書かれていない。

> リトライ処理を追加。タイムアウトを 30 秒に変更。ログ出力を修正。テストを追加。

良い例。課題の説明から始まり、各変更が理由とともにつながっている。

> 外部 API が一時的に 503 を返すと、バッチ処理全体が失敗する問題を修正する。失敗時に最大 3 回リトライする処理を追加し、リトライの待ち時間を確保するためタイムアウトを 10 秒から 30 秒に延ばした。あわせて、リトライの発生回数を運用時に確認できるようログ出力を追記し、一時エラーから回復することを検証するテストを追加した。

## 書き終えたら確認すること

文章を書き終えたら、次の観点で読み返すこと。まず、段落の中に前後とつながらない孤立した文がないかを確認する。次に、理由の書かれていない規則が残っていないかを見直す。体言止めや短い命令文の連続は、文章が羅列に戻り始めている兆候であるため、目安として 3 つ以上続いている箇所がないかも確認する。さらに、箇条書きにした箇所が、本当に独立した項目の集まりであるかも確認する。ひとつでも該当する場合は、その箇所に接続詞や理由を補って書き直すこと。
