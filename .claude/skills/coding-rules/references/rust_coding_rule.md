# Rust コーディングルール

## このドキュメントの目的

本ドキュメントは、このリポジトリで Rust のコードを書く際の基本ルールをまとめたものである。新しくコードやモジュールを追加する場合も、既存のコードを修正する場合も、ここに示す方針に従うこと。

## コーディングスタイル

インデントには半角スペース 4 個を用いる。1 行の長さは原則として 80 文字以内とし、長くなる場合は適宜折り返す。コードの整形には `rustfmt` の標準設定を用いること。特別な理由がある場合に限り設定を変更してもよいが、その際は理由をコメントやドキュメントに残すこと。

## 型・変数宣言

変数宣言では、`const` および `static` を除き、左辺に型を明示しない。型推論だけでは型が定まらない場合は、イテレーターの `collect` や `parse` のように、ターボフィッシュなどを用いて右辺側で型を指定する。やむを得ず型推論が働かない場面を除き、左辺に型を書くことは避けること。

例:

```rust
let a = 0_u32;
let v = v.iter().collect::<Vec<usize>>();
let value = "42".parse::<u64>().unwrap();
```

`const` と `static` の例:

```rust
const N: usize = 100;
static FLAG: AtomicBool = AtomicBool::new(false);
```

なお、`type` による型エイリアスの宣言 (`type Alias = ...;`) は禁止する。

## use の方法

関数や型を `use` でインポートする際は、対象そのものを直接取り込むのではなく、原則としてその親モジュールまでのパスを指定すること。たとえば `use super::front_of_house::hosting::add_to_waitlist;` のように関数を直接インポートし、本体から `add_to_waitlist();` と呼び出す書き方は禁止する。親モジュール経由で呼び出すことで、関数や型の定義位置が明確になり、ローカル定義かどうかの判別もしやすくなるためである。

例 1:

```rust
use super::front_of_house::hosting;

fn main() {
    hosting::add_to_waitlist();
}
```

例 2:

```rust
use rand::distr::{self, Distribution};

fn main() {
    let mut rng = rand::rng();
    let dist = distr::Uniform::new(0.0_f32, 1.0_f32).unwrap();
    let x = dist.sample(&mut rng);
    println!("{x}");
}
```

ローカルのモジュールを参照するときは、`crate` から始まる絶対パスではなく、`super` でたどる相対パスを使うこと。また、`use foo as bar;` のような `as` による別名付けは禁止する。

`use` を書かなくても参照できる場合であっても、本体のコードで `std::...` のようにクレートルートから始まる完全修飾パスを使うことは原則として避ける。必ず `use` で「1 つ上のモジュール」まで取り込み、`module::Item` の形で参照すること。

OK:

```rust
use std::error;

impl error::Error for FatalError {}
```

NG:

```rust
impl std::error::Error for FatalError {}
```

この「完全修飾パスを避ける」ルールは、`impl` ヘッダーだけでなく、trait bound、型注釈、関数の戻り値型、`where` 句など、型が現れるあらゆる位置に適用する。名前の衝突を避けたい場合も `as` による別名付けに頼るのではなく、取り込む単位をモジュールに揃えたうえで、`io::Error` と `error::Error` のようにモジュール経由で区別すること。

```rust
use std::{error, io};

fn f() -> io::Result<()> {
    let _ = error::Error::source;
    Ok(())
}
```

同じルートから始まる `use` が複数並ぶ場合は、可能な限り波括弧でまとめること。こうすることでインポートの見通しが良くなり、重複したパス記述も削減できる。

OK:

```rust
use std::{error, fmt, io, mem, process};
```

NG:

```rust
use std::error;
use std::fmt;
use std::io;
```

親モジュールと子モジュールの両方が必要な場合は、`self` を使って一つにまとめること。

```rust
use std::io::{self, prelude};
```

## コメントおよびドキュメント

### 通常コメント (`//`)

コメントはすべて日本語で記述すること。その際、コードの逐語訳にならないよう注意し、コードが表現する意図や、背景にあるロジックを説明する文章にすること。

配置と粒度については、変数の宣言・初期化、ループの開始、条件分岐、主要な計算ステップごとにコメントを付けるのが原則である。「コードの塊 (ブロック)」単位ではなく「論理的な 1 ステップ」単位で細かく記述し、コードを読まずにコメントだけを追っても、処理の流れとアルゴリズムが理解できる粒度を目指すこと。

記述する内容としては、なぜその初期値なのか、なぜその条件でループを抜けるのか、なぜその計算式なのかといった、数学的根拠や最適化の意図を説明する。たとえば「偶数は素数になり得ないため、3 から開始し、2 ずつ増加させる。」のような書き方である。あわせて、その行で期待している状態や、守られるべき不変条件も説明すること。たとえば「二乗計算でのオーバーフローを防ぐため、u128 にキャストして比較する。」のように書く。

### ドキュメントコメント (`///`、`//!`)

ドキュメントコメントも日本語で記述すること。クレート全体の説明は、ファイルの最上部に `//!` で記述し、クレートの概要 (機能や特徴など) を記載する。また、`trait`、`struct`、`enum`、`fn`、`mod` などのあらゆるアイテムには、公開 (pub) か非公開かを問わず、その直前に `///` で詳細なドキュメントコメントを付与すること。

関数のドキュメントコメントには、まず概要を記載し、続いて次のセクションを設けること。`# Args` セクションには引数の一覧と意味を記載し、引数や内部状態に関する制約があればその旨も明示する。`# Returns` セクションには戻り値の意味を記載する。`# Examples` セクションには、実行可能な Rust コードブロックによる使用例を記載する。なお、`# Args` と `# Returns` では、型だけでなく、その値が何を表し、どのように使われるかまで説明すること。

### エラーの書式

エラーメッセージは英語で記述すること。また、`assert_eq!` マクロでは、第 1 引数に期待値を、第 2 引数に実際の値を渡すこと。すなわち、(期待値, 実際の値) の順とする。

## ルールを満たしたコード例

以下に、すべてのルールを満たしたコード例を示す。コメントの量や文体、語尾などは、このコードを参考にして、なるべく近づけるようにコーディングすること。

```rust
//! 素数生成に関連する機能を提供するモジュールである。
//!
//! このモジュールは、指定された個数の素数をシンプルかつ確実に生成するための関数を含む。
//! 古典的な "試し割り" 法を採用し、各候補数に対して既知の素数による剰余判定を行うことで
//! 素数性を検証する。

/// 最初の `n` 個の素数を生成する。
///
/// # Args
/// - `n` - 生成する素数の個数であり、生成される素数の最大値は `u64` の範囲内に収まる必要がある。
///
/// # Returns
/// 計算された素数が昇順に格納された `Vec<u64>` を返す。
///
/// # Panics
/// - メモリの割り当てに失敗した場合 (`n` が極端に大きい場合など) にパニックする。
///
/// # Complexity
/// - 時間計算量: $O(n \sqrt{p_n})$
///   - $p_n$ は $n$ 番目の素数を表す。各素数を見つけるために、その平方根以下の素数で割り算を行う。
/// - 空間計算量: $O(n)$
///   - 見つかった素数を保持するために、$n$ に比例したメモリを使用する。
///
/// # Examples
/// ```
/// // 0 個の場合は空のベクタを返す。
/// assert_eq!(Vec::<u64>::new(), literate_primes::generate(0));
///
/// // 最初の 1 個の素数
/// assert_eq!(vec![2], literate_primes::generate(1));
///
/// // 最初の 5 個の素数
/// assert_eq!(vec![2, 3, 5, 7, 11], literate_primes::generate(5));
/// ```
#[must_use]
pub fn generate(n: usize) -> Vec<u64> {
    if n == 0 {
        return Vec::new();
    }

    // 素数を格納するためのベクタを事前に確保する。
    // 素数は見つかり次第、順次このリストに追加される。
    let mut primes = Vec::with_capacity(n);

    // 最初の素数は 2 である。
    primes.push(2);

    // 次の素数の候補値。偶数は素数になり得ないため、奇数のみを対象として
    // 3 から開始し、2 ずつ増加させる。
    let mut candidate = 3_u64;

    // 指定された個数の素数が見つかるまでループを継続する。
    while primes.len() < n {
        // 現在の候補値が素数であるかどうかを判定するためのフラグ。
        // 以下の検証ループで割り切れる素数が見つかった場合、false に更新する。
        let mut is_prime = true;

        // 候補値が合成数であるかを判定するために、既知の素数で試し割りをする。
        // 候補値の平方根以下の素数のみを確認すれば十分である。
        // なぜならば、これより大きな素数で割り切れる場合、その商は必ず平方根より小さくなり、
        // すでにチェック済みのためである。
        for &prime in &primes {
            // 素数の二乗が候補値を超えた場合、これ以上チェックする必要はない。
            // 注意: 二乗計算でのオーバーフローを防ぐため u128 にキャストして比較する。
            let p = prime as u128;
            if p * p > candidate as u128 {
                break;
            }

            // 割り切れる場合、候補値は素数ではない。
            if candidate % prime == 0 {
                is_prime = false;
                break;
            }
        }

        // 素数であると判定された場合、リストに追加する。
        if is_prime {
            primes.push(candidate);
        }

        candidate += 2;
    }

    primes
}
```

```rust
//! 郵送先情報を扱うためのデータ構造を提供するモジュールである。
//!
//! このモジュールは、住所を表す構造体と、その住所を含む配送先情報の構造体を定義する。
//! 単一の責務を持つ小さな構造体を組み合わせることで、意味の明確なデータ表現を実現する。

/// 郵送先の住所を表す。
#[derive(Debug, Clone, PartialEq, Eq)]
pub struct Address {
    /// 都道府県名
    prefecture: String,

    /// 市区町村名
    city: String,

    /// 番地、建物名、部屋番号などを含む詳細な住所
    street_address: String,

    /// 郵便番号
    postal_code: String,
}

/// 荷物の配送先情報を表す。
#[derive(Debug, Clone, PartialEq, Eq)]
pub struct ShippingLabel {
    /// 受取人の氏名
    recipient_name: String,

    /// 宛先として使用する住所
    destination: Address,

    /// 配達時に参照する連絡先電話番号
    phone_number: String,
}
