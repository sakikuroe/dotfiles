# Git コミットメッセージの記述ルール

本ドキュメントは、チームで使用する Git コミットメッセージの記述ルールを定めるものである。

## 書式

メッセージは動詞の原形 (命令形) から始める。`Fixes` (三人称単数現在) や `Fixed` (過去形) のような形は用いず、`Fix` とする。動詞の直後にはコロン (`:`) を置き、変更内容は日本語で記述する。メッセージの末尾にピリオド (`.`) はつけない。また、コミットメッセージは 1 行の要約 (Subject) のみで完結させ、本文 (Body) は原則として使用しない。

## 内容

メッセージの先頭には、変更内容を明確に示すため、原則として次に挙げる動詞のいずれかを大文字で始まる形で用いる。

- `Add`: 新機能、ファイル、テスト等の新規追加
- `Fix`: バグ、タイポ、不適切な動作の修正
- `Update`: ドキュメント、コメント、依存関係の更新 (本体の動作修正には用いない)
- `Remove`: 不要なコード、ファイル、機能の削除
- `Improve`: パフォーマンス、可読性、アーキテクチャ等の改善
- `Move`: ファイルやコードの移動
- `Rename`: ファイル名、変数名等の名称変更
- `Use`: 既存の実装から別の実装への切り替え
- `Allow`: 制約の解除や機能の有効化

`Refactor`、`Change`、`Modify` のような抽象的な動詞は避け、上記の具体的な動詞を使い分けることで、変更の意図を明確にする。

固有名詞やコード片の書式は次のように統一する。固有名詞 (例: `React`、`TypeScript`) は公式表記に従い、ファイル名・関数名・コード片はバッククォート (`` ` ``) で囲む。アクロニム (例: `API`、`URL`、`DOM`) は原則として大文字で記述する。

メッセージ中で英語表現を用いる場合、可算名詞は原則として複数形とする (例: `Add validation tests`)。ただし、明確に単一の対象を指す場合に限り、単数形と冠詞 `a` を用いる (例: `Fix a memory leak`)。冠詞 (`a`、`an`、`the`) は意味が通じる限り省略し、`doc` や `conf` のような非公式な略語は避けて、`documentation` や `configuration` のように記述する。

## 運用

1 つのコミットには、1 つの論理的な変更のみを含める。例えば、バグ修正とリファクタリングが同時に発生した場合は、それぞれ別のコミットに分割する。また、手動で絵文字を追加したり、`Merge` や `Revert` など Git が自動付与する接頭辞をメッセージに含めたりしてはならない。

## メッセージ例

- Add: `production` 環境用のビルドスクリプトを追加
- Fix: `SessionManager` のメモリーリークを修正
- Update: `README.md` に新しいセットアップ手順を追記
- Remove: 未使用の画像アセットを削除
- Improve: `search()` 関数のパフォーマンスを改善
- Move: 認証ロジックを `auth` モジュールへ移動
- Rename: コンポーネント名を `UserList` から `UserTable` に変更
- Use: リクエスト処理を Fetch API から `axios` に切り替え
- Allow: 特定オリジンからの CORS リクエストを許可
- Fix: ドキュメント中のタイポを修正
