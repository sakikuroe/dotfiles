# Issue の作成

## 概要

ユーザーの要求をもとに GitHub Issue を作成する。Issue は以降のすべての工程の基準となるため、要求と完了条件を過不足なく記述すること。特に、ユーザーが明示していない要件やスコープを想像で補うことは厳禁であり、草案を作成する前に必ず完了条件をユーザーと合意すること。

## 手順

まず、ユーザーから「やりたいこと」または「起きている問題」を受け取り、Issue の種別 (`機能追加・機能改善` / `バグ修正`) を判断する。入力から種別が明らかな場合は AI Agent が判断してよいが、判断に迷う場合はユーザーに確認すること。

草案を作成する前に、必要な情報をユーザーに確認する。完了条件は必ず確認し、背景・制約・スコープに不明な点があれば、あわせて確認しておくこと。また、既存の Issue や PR と重複するおそれがある場合は検索して確かめ、内容が 1 つの Issue に収まらない場合は分割案を提示すること。

確認が済んだら、ユーザーの回答をもとに草案を作成し、タイトル案と本文案をまとめてユーザーに提示する。本文案は `/tmp` 配下にマークダウンで記述したうえで、その中身をユーザーに見せること。本文は「Issue 本文の書き方」の種別テンプレートから始め、必要に応じて節を追加する。記述してよいのはユーザーから確認できた情報のみである。

提示した草案について、ユーザーの承認を得るまで修正と再提示を繰り返す。承認を得てから、`bash .claude/skills/issue-planning/scripts/create_issue.sh` を実行して Issue を作成する。

コマンド例:

```bash
cat <<'EOF' > /tmp/issue_body.md
## 提案する機能の概要

...

## 何をもって完了とするか

- [ ]
EOF

bash .claude/skills/issue-planning/scripts/create_issue.sh "<タイトル>" /tmp/issue_body.md
```

作成後は、結果 (URL、番号、本文) が意図どおりであることを確認する。なお、Issue 本文の内容を後から更新する場合は、`bash .claude/skills/issue-planning/scripts/update_issue_body.sh <issue番号> <body_file>` を使うこと。

## テンプレート

Issue の種別に応じて、次のテンプレートを使用する。機能追加・機能改善の場合は [templates/feature.md](./templates/feature.md) を、バグ修正の場合は [templates/bug.md](./templates/bug.md) を用いること。

## 記述上のルール

Issue は、要求や完了条件が第三者にも読み取れる状態にすること。そのために、タイトルおよび本文は日本語で書き、本文の構成は `writing-rules` スキルの `prose_structure.md` に従うこと。また、関係者の合意なしに Issue のスコープを広げてはならない。判断に迷う場合は作業を中断し、ユーザーに報告・相談すること。

## 何をもって完了とするか

- [ ] Issue が GitHub 上に作成されている。
