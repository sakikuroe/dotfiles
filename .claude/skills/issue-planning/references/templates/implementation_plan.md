## 実装方針

<!--
本コメントは、Issue の担当者が実装に着手する前に記述する。記述後、実装方針に対する確認（👍 リアクションまたは承認コメント）を得てから着手すること。

本コメントの目的は「何をどう変えるつもりか」を事前に共有し、手戻りを防ぐことにある。したがって、変更の全体像と方向性が伝わる程度の粒度で書けばよい。
以下の情報は本コメントの対象外であり、それぞれ適切な場所で扱う。

- 変更対象ファイルの詳細な一覧、テスト方法の詳細 → PR の説明文で記述する
- 技術選定や設計上の判断（選択肢の比較、不採用理由） → ADR（docs/adr/）として記録する。判断の比較が必要な時点で ADR を書くべきサインである
- 機能の背景・動機・完了条件 → Issue 本文に記述済みのため、ここでは繰り返さない

本コメントに記述する内容が長くなる場合（目安として 20 行を超える場合）、それは ADR や Design Doc（docs/design/）として独立した文書にすべき兆候である。その場合は本コメントの代わりに ADR または Design Doc の PR を作成し、レビューを経てから実装に着手する。
-->

### 何をどのように変更するか

<!--
本セクションには、この Issue に対してどのような変更を行うかの概要を記述する。

「何を」「どのように」変えるかを、2〜5 行程度で簡潔に記述すること。
読み手（確認者）がこのセクションだけで変更の方向性を把握でき、「その方針で進めてよい」と判断できる粒度が目安である。

この時点では実装に着手していないため、最終的な実装と細部が異なっていても構わない。重要なのは方向性の合意である。

以下の例を参考にすること。

適切な例:

  ```md
  GitHub Actions の matrix strategy を用いてテストスイートを 4 分割し、並列実行する。
  分割の単位はディレクトリー単位とし、各グループの実行時間が均等になるよう
  事前計測の結果を基に振り分ける。
  全ジョブの結果は GitHub のステータスチェックに集約する。
  ```

  この例では、何をするか（テストを 4 分割して並列実行）、どう分割するか（ディレクトリー単位、実行時間基準で均等化）、結果をどう扱うか（ステータスチェックに集約）が簡潔に示されている。確認者はこの 4 行で方針の妥当性を判断でき、「進めてよい」と回答できる。

不適切な例（情報が不足している）:

  ```md
  CI を並列化する。
  ```

  何をどう並列化するのかが書かれていない。確認者は「どのツールで？」「何分割？」「分割の基準は？」と確認しなければならず、確認フェーズとして機能しない。

不適切な例（詳細すぎる）:

  ```md
  GitHub Actions の matrix strategy を用いてテストスイートを 4 分割する。
  `.github/workflows/test.yml` に matrix 設定を追加し、`scripts/split_tests.py` を
  新規作成してテストファイルを振り分ける。split_tests.py は pytest の
  `--collect-only` で取得したテスト一覧を……（以下 30 行続く）
  ```

  ファイルパスやスクリプトの内部仕様は PR の説明文で扱う内容であり、本コメントでは不要である。方針の確認に必要な粒度を超えている。
-->

### 着手前に確認・相談したいこと

<!--
（任意）方針の中で判断に迷っている点、着手前に確認・相談したい点があれば記述する。
記載がなければ本セクションごと削除してよい。

このセクションの目的は、担当者が抱えている不確実性を確認者と共有し、着手前に解消することにある。着手後に発覚すると手戻りが大きい懸念点を優先的に記載すること。

以下の例を参考にすること。

適切な例:

  ```md
  テストの分割基準について、ディレクトリー単位ではなく実行時間の実測値に基づいて
  動的に振り分ける方法もある。今回はディレクトリー単位の静的な振り分けで進める
  つもりだが、将来的な偏りの発生を考慮すると動的振り分けの方がよいか？
  ```

  何について迷っているか（分割基準の選択）、自分の現時点の判断（静的振り分け）、懸念点（将来的な偏り）が明確に示されている。確認者は「今は静的でよい、偏りが出たら別 Issue で対応しよう」のように具体的に回答できる。

適切な例（他チームへの影響確認）:

  ```md
  並列化によってジョブ数が 4 倍になるため、CI ランナーの同時実行枠を超える
  可能性がある。インフラチーム（@infra-team）に枠の確認をしてから着手したい。
  ```

  着手前に解消すべきブロッカーが明示されており、誰に確認すべきかも記載されている。

不適切な例:

  ```md
  特になし。
  ```

  記載がない場合はセクションごと削除すること。「特になし」と書く必要はない。
-->
