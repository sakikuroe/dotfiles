## 本機能を提案する背景と動機

<!--
本セクションには、この機能を提案するに至った背景と動機を記述する。テンプレート全体の中で最も重要なセクションであり、ここが不十分であると対応の要否や優先度を判断できない。

「誰が」「どのような場面で」「何に困っているか」を軸に、現在の状況とその状況を変える必要がある理由を記載すること。
作業の所要時間、発生頻度、問い合わせ件数など、課題の深刻さを裏付ける定量的な情報があれば積極的に含める。
なお、実装方法はここでは扱わず、あくまで利用者の視点から課題と期待を述べること。

以下の例を参考にすること。

適切な例:

  ```md
  CI パイプラインにおけるテストの実行時間が、開発チーム全体のボトルネックになっている。現在、約 800 件のテストケースが単一のジョブで逐次実行されており、1 回の CI 実行に平均 25 分を要する。

  チームが半年前の 5 名から 12 名に拡大した結果、1 日あたりの PR 数は 3〜4 件から 10 件以上に増加したが、CI の処理能力は当時のまま据え置かれている。PR を push してからテスト結果が返るまでの待ち時間が開発者のコンテキストスイッチを誘発しており、マージまでのリードタイムも悪化傾向にある。直近 1 か月の計測では、PR のオープンからマージまでの所要時間の中央値が 4.5 時間であり、そのうち約 60% を CI の待ち時間が占めている。
  ```

  この例では、現在の状況（800 件のテストを逐次実行、平均 25 分）、状況が悪化した経緯（チーム拡大に対して CI が据え置き）、定量的な影響（リードタイム 4.5 時間、うち 60% が CI 待ち）が段落を通じて展開されている。読み手はこのセクションだけで課題の深刻さと対応の緊急度を判断できる。

不適切な例:

  ```md
  CI が遅いので並列化したい。
  ```

  この一文には背景も動機も含まれていない。どの程度遅いのか、それが誰にどのような影響を与えているのか、なぜ今対応が必要なのかが読み取れない。実装担当者がこの Issue を受け取っても、優先度を判断する材料がなく、状況の確認から始めなければならない。
-->

## 本機能の詳細と期待する動作

<!--
本セクションには、追加または変更する機能の具体的な内容と、期待する動作を記述する。前セクション（背景と動機）が「なぜ必要か」を述べるのに対し、本セクションでは「何をどうするのか」を述べる。

記述の視点は一貫して利用者側に置くこと。操作や処理の流れに沿って、何をしたら何が起きるかを具体的に記述する。
使用するライブラリ、データベーススキーマの変更、API の内部設計といった実装の詳細は本セクションの対象外であり、PR や設計ドキュメントで扱う。
通常の動作だけでなく、境界的な条件（一部の入力が欠落している場合、処理対象が存在しない場合等）における振る舞いも、想定できる範囲で記述しておくと実装時の判断に迷いが生じにくい。
本対応が既存の機能や動作に影響を及ぼす可能性がある場合は、その範囲と内容も併せて記述すること。既存の内部構造を変更する場合は、影響を受ける機能やモジュールを明記しておくとレビューやテストの計画に役立つ。

以下の例を参考にすること。

適切な例:

  ```md
  テストスイートをモジュール単位で複数のグループに分割し、CI 上で並列に実行する。開発者から見た操作（PR の push → テスト結果の確認 → マージ）は従来と同一であり、変わるのは結果が返るまでの所要時間である。

  各ジョブの実行結果は全ジョブの完了後にひとつのレポートに集約され、PR のステータスチェックとして表示される。いずれか 1 つのジョブが失敗した場合、PR 全体のステータスを失敗とする。全ジョブが成功した場合にのみ、ステータスを成功とする。
  なお、テストの実行順序が従来と変わるため、順序に依存するテスト間の暗黙的な結合が顕在化する可能性がある。また、並列ジョブ数の増加に伴い CI ランナーの使用量が増えるため、コストへの影響を確認する必要がある。
  ```

  この例では、主要な動作（テスト分割 → 並列実行 → 結果集約）が冒頭で示された後、成功・失敗時のステータス表示、既存動作への影響（実行順序の変化、コスト増）が順に述べられている。利用者の視点から一貫して記述されているため、実装担当者は変更後の挙動と影響範囲を具体的にイメージできる。

不適切な例:

  ```md
  pytest-xdist で 4 並列にする。GitHub Actions の matrix strategy を使ってジョブを分割する。
  ```

  この記述は使用するツール名を挙げているだけで、利用者から見た動作が含まれていない。テスト結果がどのように集約・表示されるのか、一部が失敗した場合にどうなるのか、既存の動作にどのような影響があるのかが分からない。実装方法は PR で扱う内容であり、本セクションには期待する動作と既存機能への影響を記述すること。
-->

## 本機能では取り扱わないこと

<!--
（任意）本機能の対応範囲から意図的に除外する内容を記述する。記載しておくことで、レビュー時や実装中に「これも対応すべきではないか」という議論が発生することを事前に防止できる。除外した項目に後日対応が必要な場合は、別 Issue を作成してリンクすること。

以下の例を参考にすること。

適切な例:

  ```md
  テストケース自体の削減や統廃合は本 Issue では扱わない。実行時間の短縮はあくまで並列化によって実現し、テスト内容の見直しは別途検討する。

  また、ローカル開発環境でのテスト並列化も対象外とする。本 Issue は CI パイプライン上での実行に限定する。
  ```

  何を除外しているか（テストの削減、ローカル環境の並列化）が具体的に述べられており、読み手は対応範囲の境界を正確に把握できる。

不適切な例:

  ```md
  関係ないことはやらない。
  ```

  何を「関係ない」と判断したのかが示されておらず、範囲の限定として機能していない。
-->

## 何をもって完了とするか

<!--
本 Issue の対応が完了したと判断するための基準を、検証可能な形で記述する。「問題なく動作する」「パフォーマンスが改善される」のような曖昧な表現は避け、第三者が客観的に合否を確認できる条件とすること。

以下の例を参考にすること。

適切な例:

  ```md
  - [ ] テストスイートが 4 並列以上で実行され、全体の所要時間が 10 分以内に収まる
  - [ ] 全ジョブの結果が 1 つのステータスチェックに集約され、PR 上で確認できる
  - [ ] 既存のテストがすべて通過する（並列化に起因する新たな失敗が発生しない）
  - [ ] 並列化後の CI ランナー使用量とコストの試算が完了している
  ```

  各項目が具体的かつ検証可能であり、第三者がチェックリストとして使用できる。数値基準（4 並列以上、10 分以内）が含まれているため、達成の判断に曖昧さが生じない。

不適切な例:

  ```md
  - [ ] CI が速くなる
  - [ ] テストが正常に動作する
  ```

  「速くなる」がどの程度を指すのか、「正常に動作する」が何を意味するのかが定義されていない。完了の判断が記述者の主観に依存してしまい、基準として機能しない。
