# 実装方針のレビュー

## 概要

Issue に着手する前に、AI Agent が実装方針を Issue コメントとして投稿し、ユーザーの承認を得る。コードを大量に生成した後で方針の誤りに気づき、手戻りが発生する事態を防ぐことが目的である。なお、Issue 本文は変更せず、方針はコメントとして追記する。

## 手順

まず Issue を再読し、目的・完了条件・スコープを把握したうえで、実装方針を検討する。具体的には、採用するアプローチとその選定理由を整理するとともに、代替案を洗い出し、それらを不採用とする理由を明確にする。あわせて、変更が必要なファイルやモジュールを特定し、完了条件をどのように検証するかも考えておく。

方針が固まったら、実装方針コメントの草案を作成する。ただし、草案の作成に先立って `templates/implementation_plan.md` を読むこと。そのうえで、テンプレートの各節を埋めて一時ファイルに書き出す。コマンド例は次のとおりである。

```bash
cat <<'EOF' > /tmp/plan.md
## 実装方針
...
## 変更対象
...
## 検討した代替案と不採用理由
...
## テスト方針
...
EOF
bash .claude/skills/issue-planning/scripts/post_implementation_plan.sh <issue番号> /tmp/plan.md
```

草案を投稿したら、進捗コメントで状態を `方針レビュー待ち` に記録する。この投稿には `bash .claude/skills/issue-planning/scripts/add_progress_comment.sh <issue番号> <body_file>` を用いる。

その後は、ユーザーの承認または修正依頼を待つ。承認が得られたら、作業ブランチの作成へ進む。一方、修正依頼があった場合は方針を見直し、コメントを投稿し直す。

## 原則

Issue 本文は変更せず、方針はコメントとして追記すること。また、方針が固まる前にブランチを切ってはならず、ブランチの作成は方針承認後に行う。修正依頼があった場合は、新しいコメントを投稿し直し、再び承認を待つこと。判断に迷う場合は作業を中断し、ユーザーに報告・相談すること。

## この段階の完了条件

この段階は、次の状態がすべて満たされたときに完了となる。すなわち、実装方針コメントが Issue に投稿され、進捗に `状態: 方針レビュー待ち` が記録されていること。そのうえで、ユーザーから承認が得られており、作業ブランチの作成に進める状態になっていることである。
