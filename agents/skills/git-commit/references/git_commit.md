# Git コミットメッセージの記述ルール

本ドキュメントは, チームで使用する Git コミットメッセージの記述ルールである.

## 書式

1. 命令形で記述する.
    メッセージは, 動詞の原形 (命令形) から始める. `Fixes` (三単現) や `Fixed` (過去形) のような形は用いず, `Fix` とする.
    動詞の直後にコロン (`:`) を置き, 変更内容は日本語で記述する.

2. 末尾にピリオドをつけない.
    メッセージの末尾には, ピリオド (`.`) をつけない.

3. メッセージは1行で完結させる.
    コミットメッセージは1行の要約 (Subject) のみとし, 本文 (Body) は原則として使用しない.

## 内容

1. 文頭に, 大文字で始まる特定の動詞を用いる.
    メッセージの先頭には, 変更内容を明確に示すため, 原則として次に挙げる動詞のいずれかを用いる.
    - `Add`: 新機能, ファイル, テスト等の新規追加.
    - `Fix`: バグ, タイポ, 不適切な動作の修正.
    - `Update`: ドキュメント, コメント, 依存関係の更新 (本体の動作修正には用いない).
    - `Remove`: 不要なコード, ファイル, 機能の削除.
    - `Improve`: パフォーマンス, 可読性, アーキテクチャ等の改善.
    - `Move`: ファイルやコードの移動.
    - `Rename`: ファイル名, 変数名等の名称変更.
    - `Use`: 既存の実装から別の実装への切り替え.
    - `Allow`: 制約の解除や機能の有効化.

2. 固有名詞やコード片の書式を統一する.
    - 固有名詞 (例: `React`, `TypeScript`) は公式表記に従う.
    - ファイル名, 関数名, コード片はバッククォート (`` ` ``) で囲む.
    - アクロニム (例: `API`, `URL`, `DOM`) は原則として大文字で記述する.

3. 名詞は原則として複数形を用いる.
    可算名詞は原則として複数形を用いる (例: `Add validation tests`). ただし, 明確に単一の対象を指す場合にのみ, 単数形と冠詞 `a` を用いる (例: `Fix a memory leak`).

4. 冠詞や略語を適切に扱う.
    - 冠詞 (`a`, `an`, `the`) は, 意味が通じる限り省略する.
    - `doc` や `conf` のような非公式な略語は避け, `documentation` や `configuration` のように記述する.

5. 抽象的な動詞を避ける.
    `Refactor`, `Change`, `Modify` のような抽象的な動詞は避け, ルール 4 に示す具体的な動詞を使い分けることで変更の意図を明確にする.

## 運用

1. 1コミット1関心事の原則を守る.
    1つのコミットには, 1つの論理的な変更のみを含める. 例えば, バグ修正とリファクタリングは別のコミットに分割する.

2. 禁止事項.
    手動で絵文字を追加したり, `Merge` や `Revert` など Git が自動付与する接頭辞をメッセージに含めたりしない.

## メッセージ例

- Add: `production` 環境用のビルドスクリプトを追加した
- Fix: `SessionManager` のメモリーリークを修正した
- Update: `README.md` に新しいセットアップ手順を追記した
- Remove: 未使用の画像アセットを削除した
- Improve: `search()` 関数のパフォーマンスを改善した
- Move: 認証ロジックを `auth` モジュールへ移動した
- Rename: コンポーネント名を `UserList` から `UserTable` に変更した
- Use: リクエスト処理を Fetch API から `axios` に切り替えた
- Allow: 特定オリジンからの CORS リクエストを許可した
- Fix: ドキュメント中のタイポを修正した
