# Issue の作成

## 概要

ユーザーの要求をもとに GitHub Issue を作成する.
Issue は以降のすべての Step の基準となるため, 要求・完了条件・進捗を過不足なく記述すること.
草案を作成する前に完了条件をユーザーと合意し, ユーザーが明示していない要件を想像で補わないこと.

## 手順

- ユーザーから「やりたいこと」または「起きている問題」を受け取る.
- Issue の種別 (`機能追加・機能改善` / `バグ修正`) を判断する. 入力から明らかな場合は AI Agent が判断してよい. ただし, 判断に迷う場合はユーザーに確認すること.
- 草案を作成する前に, 必要な情報をユーザーに確認すること.
    - 完了条件は必ず確認すること. ユーザーが明示していない要件やスコープを想像で補わないこと.
    - また, 背景・制約・スコープで不明な点があれば合わせて確認する.
    - ただし, 質問は 1〜3 点に絞り, 優先度の高いものから尋ねること. 自由記述より選択肢の提示が適切な場合は選択肢を示す.
- 既存 Issue / PR との重複のおそれがあれば検索して確認する. 1 Issue に収まらない場合は分割案を提示すること.
- ユーザーの回答をもとに草案を作成し, タイトル案と本文案をまとめてユーザーに提示する.
    - 本文は「Issue 本文の書き方」の種別テンプレートから始め, 必要に応じて節を追加する.
    - ユーザーから確認した情報のみを記述すること. 想像で要件を補わないこと.
- ユーザーの了承を得るまで修正と再提示を繰り返す. Issue を作成してから内容を詰めるのではなく, 作成前に本文を合意済みにすること.
- 草案が固まったら, 次の点を最終確認する.
    - タイトルだけで第三者に内容が伝わるか.
    - `完了条件` が客観的に判定でき, 曖昧語を含んでいないか.
    - `進捗` に `ブランチ` と `PR` の初期値が入っているか.
- ユーザーの認証を得てから `bash ${CLAUDE_SKILL_DIR}/scripts/create_issue.sh` を実行する.
    - 実行場所はメインリポジトリーでも worktree でも構わない.
    - 本文はヒアドキュメントで一時ファイルに書き出してから渡すこと. スクリプトが `--body-file` を必須としているため, シェル内でバッククォートをエスケープする事故を防げる.
    - コマンド例:
        ```bash
        cat <<'EOF' > /tmp/issue_body.md
        ## 機能の概要

        ...

        ## 進捗

        - 状態: 未着手
        - ブランチ: 未作成
        - PR: 未作成
        - 次: Step 03 (作業ブランチと worktree 作成)
        EOF

        bash ${CLAUDE_SKILL_DIR}/scripts/create_issue.sh "<タイトル>" /tmp/issue_body.md
        ```
- 作成後は結果 (URL, 番号, 本文) が意図どおりであることを確認する. なお, Issue 本文を後から更新する場合は `bash ${CLAUDE_SKILL_DIR}/scripts/update_issue_body.sh <issue番号> <body_file>` を使うこと. 進捗フィールドの更新でも同じスクリプトを使う.

## Issue 本文の書き方

以下の種別テンプレートは最大構成であり, 該当しない節は見出しごと削除して使う.
緊急度の低い小さな Issue ほど節を減らしてよく, 「追加する背景と動機」「根本原因」「採用するアプローチと判断根拠」などは起票時に空のまま残して作業中に追記すればよい.

### 書き方の指針

#### Issue に書くべきこと

コードやドキュメントからは復元できない情報を書くこと. 具体的には, 「なぜこの判断をしたか」「何を検討して何を捨てたか」「どう検証したか」というプロセスと文脈, またテスト実行コマンドや計測結果など, 作業に紐づく一時的だが有用な情報が対象となる.

#### Issue に書かないこと

API の使い方, 設定値の意味, アーキテクチャの全体像など, コードコメントやドキュメントに書けば済む情報は Issue に書かないこと. 二重管理になり陳腐化する.

#### 却下した案こそ書く

意思決定を将来見直すとき, 判断の根拠は決定そのものより価値を持つ. 「A を選んだ」だけでなく, 「B と C を検討して A にした理由」を残すこと.

上記に加えて, 次の指針に従うこと.

- 見出しは「何を書く節か」がそのまま伝わる表現にすること. 短い抽象語だけの見出しは避ける.
- `完了条件` は `- [ ]` 形式で, 客観的に判定できる成果だけを書くこと. 曖昧語を入れない.
- 実装手段だけで終わらせず, 要求や完了条件が第三者にも読み取れる状態にすること.

### テンプレート

種別が決まったら対応するファイルを読む.

- 機能追加・機能改善 → [templates/feature.md](./templates/feature.md)
- バグ修正 → [templates/bug.md](./templates/bug.md)

## 原則

- Issue のタイトルおよび本文は日本語で書くこと.
- Issue 本文は `writing-rules` スキルの `prose_structure.md` に従うこと.
- ユーザーの認証を得ずに `create_issue.sh` を実行しないこと.
- 関係者の合意なしに Issue のスコープを広げないこと.
- 判断に迷う場合は作業を中断し, ユーザーに報告・相談すること.

## この phase の完了条件

- [ ] Issue が GitHub 上に作成されている.
- [ ] Issue 本文に客観的に判定できる `完了条件` と `進捗` が含まれている.
- [ ] Step 03 を開始できる状態になっている.
