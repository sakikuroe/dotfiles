# Issue の作成

## 概要

ユーザーの希望をもとに GitHub Issue を作成します.
Issue は以降の全 Step の基準となるため, 要求, 完了条件, 進捗を過不足なく書きます.
質問を並べず草案を早く出し, ユーザーの確認で仕上げます.

## 手順

- ユーザーから「やりたいこと」または「起きている問題」を受け取ります.
- Issue の種別を判断します.
    - `機能追加` / `バグ修正` / `改善` のいずれかです.
    - 入力から明らかなら AI Agent が決めて構いません. 迷う場合だけ確認します.
- 草案に必要な情報が不足している場合のみ, 1〜3 点に絞って確認します.
    - 書ける情報が揃っていれば追加質問はしません.
    - 自由記述より選択肢を提示したほうが早い場合はそうします.
- 既存 Issue / PR と重複のおそれがあれば検索して確認します.
    - 1 Issue に収まらない場合は分割案を提示します.
- 草案を作成し, ユーザーへ提示します.
    - タイトル案と本文案をまとめて一度に示します.
    - 本文は「Issue 本文の書き方」の種別テンプレートから始め, 必要に応じて節を足します.
- ユーザーの了承を得るまで修正と再提示を繰り返します.
    - Issue を作成してから内容を詰めるのではなく, 作成前に本文を合意済みにします.
- 最終確認を行います.
    - タイトルだけで第三者に内容が伝わるか.
    - `完了条件` が客観的に判定でき, 曖昧語を含んでいないか.
    - `進捗` に `ブランチ` と `PR` の初期値が入っているか.
- ユーザーの認証後に `bash ${CLAUDE_SKILL_DIR}/scripts/create_issue.sh` を実行します.
    - 実行場所: メインリポジトリでも worktree でもどちらでも可.
    - 本文はヒアドキュメントで一時ファイルに書き出してから渡します. スクリプトが `--body-file` を強制するため, シェル内で本文に含まれるバッククォートをエスケープする事故を防げます.
    - コマンド例:
        ```bash
        cat <<'EOF' > /tmp/issue_body.md
        ## 機能の概要

        ...

        ## 進捗

        - 状態: 未着手
        - ブランチ: 未作成
        - PR: 未作成
        - 次: Step 03 (作業ブランチと worktree 作成)
        EOF

        bash ${CLAUDE_SKILL_DIR}/scripts/create_issue.sh "<タイトル>" /tmp/issue_body.md --label "<label>"
        ```
- 作成結果 (URL, 番号, 本文) が意図どおりか確認します.
- Issue 本文を後から更新する場合は `bash ${CLAUDE_SKILL_DIR}/scripts/update_issue_body.sh <issue番号> <body_file>` を使います. 進捗フィールドの更新でも同じスクリプトを使います.

## Issue 本文の書き方

以下の種別テンプレートは最大構成です. 該当しない節は見出しごと削除して使います.
小さく緊急度の低い Issue ほど, 節も少なくして構いません.
「背景・動機」「原因」「検証メモ」などは起票時に空でよく, 作業中に追記します.

### 書き方の指針

#### Issue に書くべきこと

コードやドキュメントからは復元できない情報. 「なぜこの判断をしたか」「何を検討して何を捨てたか」「どう検証したか」のプロセスと文脈. テスト実行コマンドや計測結果など, 作業に紐づく一時的だが有用な情報.

#### Issue に書かないこと

API の使い方, 設定値の意味, アーキテクチャの全体像など, コードコメントやドキュメントに書けば済むもの. Issue に書くと二重管理になり腐る.

#### 却下した案こそ書く

意思決定を将来見直すとき, 判断の根拠は決定そのものより価値を持つ. 「A を選んだ」より「B と C を検討して A にした理由」を残す.

上記に加えて, 次の指針に従います.

- 見出しは「何を書く節か」がそのまま伝わる表現にします. 短い抽象語だけの見出しは避けます.
- `完了条件` は `- [ ]` 形式で, 客観的に判定できる成果だけを書きます. 曖昧語を入れません.
- 実装手段だけで終わらせず, 要求や完了条件が第三者にも読み取れる状態にします.

### テンプレート

種別が決まったら対応するファイルを読みます.

- 機能追加 → [templates/feature.md](./templates/feature.md)
- バグ修正 → [templates/bug.md](./templates/bug.md)
- 改善     → [templates/improvement.md](./templates/improvement.md)

## 原則

- Issue のタイトルおよび本文は日本語で書きます.
- ユーザー認証なしに `create_issue.sh` を実行しません.
- 関係者の合意なしに Issue のスコープを広げません.
- 判断に迷う場合は作業を中断し, ユーザーに報告, 相談します.

## この phase の完了条件

- [ ] Issue が GitHub 上に作成されている.
- [ ] Issue 本文に客観的に判定できる `完了条件` と `進捗` が含まれている.
- [ ] Step 03 を開始できる状態になっている.
