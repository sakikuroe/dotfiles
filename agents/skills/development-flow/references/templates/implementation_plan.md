## 実装方針

<!--
「何をどう変更するか」だけでなく,「なぜそのアプローチを選んだか」まで書くこと.
アプローチの選択理由がないと, ユーザーはレビューで「なぜこの方法？」と確認しなければならず,
方針レビューの意味が薄れる.

書くべき内容:
- 変更の大まかな流れ (例: 既存の A を拡張して B を実現する)
- 採用した理由 (例: C という代替もあるが, 既存の仕組みに沿っているため A の拡張を選んだ)

適切な例:

  ```md
  ## 実装方針
  `01_sync_main.md` の再開判定テーブルに条件分岐を追加し, Issue コメント履歴を参照して
  「方針承認済み / 未投稿」を区別する.
  Issue コメントのパース処理は `gh issue view` の JSON 出力を使い, `jq` でフィルタリングする.
  スクリプト化はせず, Step 01 のテキスト手順として記述する.
  判定ロジックが読み手に明確に伝わるよう, 条件を箇条書きで列挙する形にする.
  ```

  このように, 何を変更して (再開判定テーブルに条件追加), どう実現して (gh + jq),
  なぜその形にするか (スクリプト化しない理由) が 1 段落で読み取れる.

不適切な例:

  ```md
  ## 実装方針
  Step 01 のファイルを修正する.
  ```

  何をどう修正するのかが書かれていない.
  読み手は「具体的には？」と聞き返さなければならず, レビューのラウンドトリップが増える.
-->

## 変更対象

<!--
変更するファイルとモジュールをリストアップする.
新規作成・変更・削除の別が一目でわかるように書く.
どのような変更をするかを一言添えると, 変更規模の見当がつきやすい.

適切な例:

  ```md
  ## 変更対象
  - `references/01_sync_main.md` — 再開判定テーブルを更新 (変更)
  - `references/03_review_implementation_plan.md` — 新規作成
  - `SKILL.md` — 手順リストと状態値を更新 (変更)
  ```

  各行にファイルパスと変更内容の概要がある. 新規 / 変更の区別も付いている.

不適切な例:

  ```md
  ## 変更対象
  development-flow 配下のいくつかのファイル
  ```

  どのファイルに何をするのかが特定できない.
  レビュアーは変更の規模も影響範囲も把握できない.
-->

## 検討した代替案と不採用理由

<!--
採用しなかったアプローチと, そのアプローチを選ばなかった理由を書く.
代替案がない場合は「なし（理由）」と書き, なぜ代替を検討しなかったか (方針が自明なため, など) を添える.
却下の理由こそ意思決定の根拠であり, 将来見直す際の最大の手がかりになる.

適切な例:

  ```md
  ## 検討した代替案と不採用理由
  - `docs/` に方針文書を置いて PR でレビューする方式:
    ラインコメントが使える利点はあるが, 方針レビュー用 PR と実装本体 PR が分離して手順が増える.
    個別 Issue レベルの方針には過剰と判断し, 不採用.
  ```

適切な例 (代替なし):

  ```md
  ## 検討した代替案と不採用理由
  なし. 変更対象が 1 ファイルかつ修正箇所が 1 行であり, アプローチは自明なため.
  ```

  「なし」とだけ書くのでなく, なぜ検討不要だったかを添えている.

不適切な例:

  ```md
  ## 検討した代替案と不採用理由
  なし.
  ```

  なぜ代替を検討しなかったのかが不明. 本当に検討したのか, 単に省略しただけなのか読み取れない.
-->

## テスト方針

<!--
完了条件をどのように検証するかを書く.
「テストが通る」のような抽象的な表現は避け, 何をどう確認するかを具体的に書くこと.
自動テストがない場合は, 手動で確認する手順を書く.

適切な例:

  ```md
  ## テスト方針
  - `ls references/` で Step ファイルのリナンバーが正しく行われていることを確認する.
  - `01_sync_main.md` の再開判定を, 「方針コメントあり承認済み」「方針コメントあり未承認」
    「方針コメントなし」の 3 ケースで手動トレースし, 正しい Step に分岐することを確認する.
  ```

  確認する手順と期待結果が具体的に書かれている.
  自動テストがない場合も, どのように手動確認するかが明示されている.

不適切な例:

  ```md
  ## テスト方針
  動作確認する.
  ```

  何を確認するのかが書かれていない.
  「動作確認」が何を指すのかは人によって異なるため, 完了の判断ができない.
-->
