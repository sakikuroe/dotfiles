## 実装方針

<!--
「何をどう変更するか」だけでなく,「なぜそのアプローチを選んだか」まで書くこと.
アプローチの選択理由がないと, ユーザーはレビューで「なぜこの方法を選んだのだろうか」と確認しなければならない.

書くべき内容:
- 変更の概要 (例: 既存の A を拡張して B を実現する)
- 採用した理由 (例: C という代替もあるが, 既存の仕組みに沿っているため A の拡張を選んだ)

適切な例:

  ```md
  ## 実装方針
  GitHub Actions の matrix strategy を用いてテストスイートを 4 分割し, 並列実行する.
  既存の CI 設定ファイルへの追加のみで実現でき, 新たな依存関係やインフラの導入が不要なため,
  セルフホストランナーの追加導入よりもこの方式を採用する.
  分割の単位はディレクトリー単位とし, 各グループの実行時間が均等になるよう
  事前に計測した実行時間を基に振り分ける.
  ```

  このように, 何を変更して (matrix strategy でテストを 4 分割), どう実現して (既存 CI 設定への追加のみ),
  なぜその形にするか (セルフホストランナー導入よりコストが低い) が 1 段落で読み取れる.

不適切な例:

  ```md
  ## 実装方針
  CI の設定ファイルを修正する.
  ```

  何をどう修正するのかが書かれていない.
  読み手は「具体的には？」と聞き返さなければならず, レビューのラウンドトリップが増える.
-->

## 変更対象

<!--
変更するファイルとモジュールをリストアップする.
新規作成・変更・削除の別が一目でわかるように書く.
どのような変更をするかを一言添えると, 変更規模の見当がつきやすい.

適切な例:

  ```md
  ## 変更対象
  - `.github/workflows/test.yml` — matrix strategy を追加し, テストを 4 並列実行 (変更)
  - `scripts/split_tests.py` — テストをグループに振り分けるスクリプト (新規作成)
  - `docs/ci.md` — 並列実行時のジョブ構成を追記 (変更)
  ```

  各行にファイルパスと変更内容の概要がある. 新規 / 変更の区別も付いている.

不適切な例:

  ```md
  ## 変更対象
  CI 関連のいくつかのファイル
  ```

  どのファイルに何をするのかが特定できない.
  レビュアーは変更の規模も影響範囲も把握できない.
-->

## 検討した代替案と不採用理由

<!--
採用しなかったアプローチと, そのアプローチを選ばなかった理由を書く.
代替案がない場合は「なし（理由）」と書き, なぜ代替を検討しなかったか (方針が自明なため, など) を添える.
却下の理由こそ意思決定の根拠であり, 将来見直す際の最大の手がかりになる.

適切な例:

  ```md
  ## 検討した代替案と不採用理由
  - セルフホストランナーを追加して並列数を増やす方式:
    処理能力は向上するが, ランナーの運用・保守コストが増え, 障害時の切り分けも複雑になる.
    既存の matrix strategy で目標の実行時間を達成できる見込みのため, 不採用.
  ```

適切な例 (代替なし):

  ```md
  ## 検討した代替案と不採用理由
  なし. 既存の CI 基盤を変更せず matrix strategy を追加するだけで実現でき,
  他に検討すべき構成上の選択肢がないため.
  ```

  「なし」とだけ書くのでなく, なぜ検討不要だったかを添えている.

不適切な例:

  ```md
  ## 検討した代替案と不採用理由
  なし.
  ```

  なぜ代替を検討しなかったのかが不明. 本当に検討したのか, 単に省略しただけなのか読み取れない.
-->

## テスト方針

<!--
完了条件をどのように検証するかを書く.
「テストが通る」のような抽象的な表現は避け, 何をどう確認するかを具体的に書くこと.
自動テストがない場合は, 手動で確認する手順を書く.

適切な例:

  ```md
  ## テスト方針
  - `.github/workflows/test.yml` の matrix 設定を確認し, 4 つのジョブが並列に起動することを確認する.
  - いずれか 1 つのジョブで意図的にテストを失敗させ, PR 全体のステータスチェックが失敗になることを確認する.
  - 全ジョブが成功した場合に, ステータスチェックが成功になることを確認する.
  ```

  確認する手順と期待結果が具体的に書かれている.
  自動テストがない場合も, どのように手動確認するかが明示されている.

不適切な例:

  ```md
  ## テスト方針
  動作確認する.
  ```

  何を確認するのかが書かれていない.
  「動作確認」が何を指すのかは人によって異なるため, 完了の判断ができない.
-->
