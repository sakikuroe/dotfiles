# 日本語技術文書作成のための記述ルール

本ドキュメントは, 日本語技術文書を作成する際の記述ルールを定めたものです. 文書全体の品質と一貫性を確保するために, 本ルールに従って記述してください.

## 句読点, スペース

### 句読点

- 読点は `, ` を用います.
- 句点は `. ` を用います.
- 日本語の `、` (U+3001) と `。` (U+3002) は原則として使用しません.

読点の詳細:
- カンマ `,` (U+002C) と半角スペース ` ` (U+0020) の組み合わせ `, ` を用います.

句点の詳細:
- ピリオド `.` (U+002E) の後に半角スペース ` ` (U+0020) を 1 つ入れ, `. ` とします.
- ピリオドの後に改行する場合など, 文末にあたる場合は, 末尾の半角スペースを省略します.

正しい記述例:
- 本設定は, 管理者のみが変更できます.
- このツールは, 認証が完了していない場合にエラーを返します.

誤った記述例:
- `本設定は、管理者のみが変更できます。` (読点に `、`, 句点に `。` を使用)
- `このツールは、認証が完了していない場合にエラーを返します。` (読点に `、`, 句点に `。` を使用)

### 文末表現

記述を文と句に分類し, それぞれ異なる文末規則を適用します.

文: 述語（動詞・助動詞）をもつ記述.
- `です, ます` 体で終えます.
- 名詞止めは使用しません.

句: 述語をもたない断片的な記述（ラベル, 引数説明, 項目名など）.
- 名詞で終えます.
- `です, ます` は付けません.

良い例:
- `本設定は, 管理者のみが変更できます.` (文 → ます体)
- `内部状態のカーソル座標のみを更新する.` (文 → 動詞止め)
- `会員種別.` (句 → 名詞止め)
- `申請期限.` (句 → 名詞止め)
- `入力ループを継続するかどうかを表す状態.` (句 → 名詞止め)
- `` `timeout` - 応答待ちの上限時間. `` (句 → 名詞止め)

悪い例:
- `会員種別です.` (句に `です` を付けている)
- `入力ループを継続するかどうかを表す状態です.` (句に `です` を付けている)
- `内部状態のカーソル座標のみを更新.` (文なのに述語を省いている)

### 箇条書き

- 箇条書きやリストの各項目も, 1 文として書く場合は文末に句点を付けます.

正しい記述例:
- 本機能は, 任意のファイルを指定できます.
- 次の手順でインストールを行います.

誤った記述例:
- 本機能は, 任意のファイルを指定できます (文末に句点がない)
- 次の手順でインストールを行います (文末に句点がない)

- 体言止めや単語のみで構成される項目には, 句点を付けません.
- 同一のリスト内では, 文末表現 (句点の有無, 体言止めか文末表現か) を統一してください.

正しい記述例 (体言止めのリスト):
- ファイルの選択
- 設定の確認
- インストールの実行

誤った記述例 (文末表現が不統一, 単語のみで構成される項目に句点が付いている):
- ファイルの選択
- 設定の確認.
- インストールを実行する.

### スペース

- 日本語と英単語, または半角数字の間には必ず半角スペースを挿入します.

正しい記述例:
- ファイルサイズは 512 MB まで許容されます.
- この API バージョンは v2 です.

誤った記述例:
- ファイルサイズは512MBまで許容されます. (数字と単位の間にスペースがない)
- このAPIバージョンはv2です. (日本語と英単語, 英単語と数字の間にスペースがない)

- 以下の場合はスペースを挿入しません.
    - 句読点や引用符, 文頭, 文末に隣接する場合.
    - 単位や形式を示す記号 (%, px, °C 等) が数字に続く場合.
    - 日本語と英単語・数字が連結して 1 語を構成する固有表現の場合.

句読点や引用符に隣接する場合の例:
- 正しい記述例: ファイル名は "example.docx" です.
- 正しい記述例: 主要な機能 (例: 検索, 置換) について解説します.
- 誤った記述例: ファイル名は " example.docx " です. (引用符と単語の間に不要なスペースがある)
- 誤った記述例: 主要な機能 ( 例: 検索, 置換 ) について解説します. (括弧の内側に不要なスペースがある)

単位記号の例:
- 正しい記述例: 解像度は 300dpi です.
- 正しい記述例: CPU 使用率が 95% に達しました.
- 誤った記述例: 解像度は 300 dpi です. (数字と単位の間に不要なスペースがある)
- 誤った記述例: CPU 使用率が 95 % に達しました. (数字と単位の間に不要なスペースがある)

固有表現の例:
- 正しい記述例: Webサイトを構築します.
- 誤った記述例: Web サイトを構築します. (固有表現 "Webサイト" の中に不要なスペースがある)

- 上記のルールに当てはまらない場合や, スペースを挿入すべきか判断に迷う場合は, スペースを入れることを優先します.

## 括弧, 記号, 書式

### 括弧

- 括弧は半角の `(` `)` (U+0028, U+0029) を使います.
- 全角括弧 `（` `）` (U+FF08, U+FF09) は使用しないでください.

正しい記述例:
- この処理 (transaction) の概要を以下に示します.
- 引数 (argument) には文字列を渡します.

誤った記述例:
- `この処理（transaction）の概要を以下に示します.` (全角括弧を使用)
- `引数（argument）には文字列を渡します.` (全角括弧を使用)

### 括弧の前後

- 左括弧 `(` の前と右括弧 `)` の後ろには原則として半角スペースを挿入します.
- ただし, 文頭や数式, プログラムコードの場合はスペースを挿入しません.

正しい記述例:
- ソースコード (Python 3.10) を添付します.
- (注) この機能は非推奨です.
- `f(x) = y`

誤った記述例:
- ソースコード(Python 3.10)を添付します. (括弧の前後にスペースがない)
-  (注) この機能は非推奨です. (文頭の括弧の前に不要なスペースがある)
- `f (x) = y` (関数の一般的な表記 `f(x)` の `f` と `(x)` の間に不要なスペースがある)

### 引用符

- 引用符は半角のダブルクォーテーション `"` (U+0022) を用います.
- 全角のダブルクォーテーション `“` `”` (U+201C, U+201D) は使用しないでください.
- 引用符 `"` を強調の目的で使用しないでください.

正しい記述例:
- ファイル名は "config.json" としてください.
- この概念は "Ownership" と呼ばれています.

誤った記述例:
- `ファイル名は “config.json” としてください.` (全角のダブルクォーテーションを使用)
- この機能は "非常に" 重要です. (強調の目的で引用符を使用)

### コード, コマンドの表記

- ファイル名, コマンド名, 関数名, 変数名などをインラインで記載する場合は, バッククォート `` ` `` で囲みます.
- 複数行のコードやコマンドを記載する場合は, コードブロック (``` ``` ```) を使用し, 言語名を指定します.

正しい記述例:
- 設定ファイル `config.yaml` を編集してください.
- `pip install numpy` を実行します.

誤った記述例:
- 設定ファイル config.yaml を編集してください. (コード要素がバッククォートで囲まれていない)
- pip install numpy を実行します. (コマンドがバッククォートで囲まれていない)

## 表現


### 主語の明示

- 動作の主体が曖昧になる場合は, 主語を明示してください.
- 特に手順書では, 操作の実行者 (ユーザー, システム, 管理者など) を明確にしてください.

正しい記述例:
- ユーザーがログインボタンを押すと, システムは認証処理を開始します.

誤った記述例:
- ログインボタンを押すと, 認証処理を開始します. (主語が不明確)

### 能動態と受動態

- 原則として能動態を使用してください.
- ただし, 動作の対象を強調する場合や, 動作主体を特定する必要がない場合には, 受動態を使用できます.

正しい記述例:
- システムがログファイルを生成します. (能動態)
- ログファイルは起動時に自動的に生成されます. (動作主体の明示が不要な場合)

誤った記述例:
- ログファイルが生成されます. (動作主体が不明確で, 能動態で書ける場面)
### 二重否定の回避

- 二重否定は使用しないでください. 肯定表現で書き換えてください.

正しい記述例:
- この機能は常に有効です.

誤った記述例:
- この機能が無効にならないわけではありません. (二重否定で意味が曖昧)

### 曖昧な表現の回避

- "など", "等", "いろいろ" といった曖昧な表現は, 可能な限り具体的な内容に置き換えてください.
- やむを得ず使用する場合は, 代表的な要素を 2 つ以上列挙してから使用してください.

正しい記述例:
- Python, Ruby, JavaScript などのスクリプト言語に対応しています.

誤った記述例:
- いろいろな言語に対応しています. (具体性がない)
- スクリプト言語などに対応しています. (列挙が不十分)

## 外国語, 外来語

### 外国語の固有名詞, 技術用語

- 製品名, 固有の技術名, または日本語での定訳や定着した表記が存在しない用語については, 可能な限り元の言語で記載します.
- ただし, "コンパイル", "デバッグ" など, すでに日本語 (カタカナ語) として広く定着している用語については, カタカナで記載してください.
- 迷った場合は, JIS 規格や各分野の標準的な用語集を参照してください.

正しい記述例:
- このシステムは Dijkstra's algorithm を応用しています.
    - もしくは "このシステムは Dijkstra 法を応用しています." とします.
- このシステムは diffusion model を使用しています.
    - もしくは "このシステムは拡散モデルを使用しています." とします.
- 不要なメモリをガベージコレクションで解放します. (定着しているためカタカナ)
- このモデルは Transformer をベースにしています. (固有名詞的なアルゴリズム名のため英語)

誤った記述例:
- このシステムはダイクストラ法を応用しています. (人名がカタカナ表記になっている)
- このシステムはディフュージョンモデルを使用しています. (カタカナ表記になっている)
- 不要なメモリを garbage collection で解放します. (定着している用語を英語にしている)
- このモデルはトランスフォーマーをベースにしています. (固有名詞をカタカナにしている)

### 外来語の長音符号

- 外来語の長音符号 (ー) は, 原則として省略しないでください.
- ただし, 固有名詞としての製品名やサービス名がもともと長音符号を省略した表記を採用している場合は, その表記に従います.

正しい記述例:
- サーバーのログを確認します.
- このコンピューターのスペックは高いです.

誤った記述例:
- サーバのログを確認します. (長音符号が省略されている)
- このコンピュータのスペックは高いです. (長音符号が省略されている)

## 漢字, ひらがな, 送り仮名

### 漢字とひらがなの使い分け

- 形式名詞, 補助動詞, 副助詞は原則としてひらがなで記載してください.
- 以下に代表的な使い分けの例を示します.

| 種別 | ひらがな (正) | 漢字 (誤) |
|---|---|---|
| 形式名詞 | こと, もの, とき, ところ, わけ, ほう, うち, ため | 事, 物, 時, 所, 訳, 方, 内, 為 |
| 補助動詞 | ～してください, ～してほしい, ～してみる, ～しておく, ～していく, ～してくる | ～して下さい, ～して欲しい, ～して見る, ～して置く, ～して行く, ～して来る |
| 副詞・接続詞 | さらに, すでに, および, ならびに, または, あるいは, ただし, なお | 更に, 既に, 及び, 並びに, 又は, 或いは, 但し, 尚 |
| 副助詞等 | ～のとおり, ～のほか, ～のうえ, ～のうち, ～のもと | ～の通り, ～の他, ～の上, ～の内, ～の下 |

- ただし, 漢字が本来の意味で使われる場合は漢字で記載します.

正しい記述例:
- 設定を変更することができます. (形式名詞)
- 事の発端は昨年にさかのぼります. (本来の意味)
- ドキュメントを参照してください. (補助動詞)
- 記載のとおり実行します. (副助詞)

誤った記述例:
- 設定を変更する事ができます. (形式名詞を漢字にしている)
- ドキュメントを参照して下さい. (補助動詞を漢字にしている)
- 記載の通り実行します. (副助詞を漢字にしている)

### 送り仮名

- 送り仮名は省略しないでください.

正しい記述例:
- 定例の打ち合わせを実施します.
- 参加申し込みは本日までとします.

誤った記述例:
- 定例の打合せを実施します. (送り仮名が省略されている)
- 参加申込みは本日までとします. (送り仮名が省略されている)

### 同義語, 表記ゆれの統一

- 同一の文書内で, 同じ概念を指す語は統一してください.
- 文書の冒頭や付録に用語集を設け, 使用する表記を定義することを推奨します.

正しい記述例:
- システムが処理を実行します. (文書全体で「実行」に統一)

誤った記述例:
- システムが処理を実行します. (中略) システムが処理を行います. (「実行」と「行う」が混在)

## 用語

### 半角カタカナ, 全角英数字, 全角の ASCII 記号

- 半角カタカナは使用しないでください.
- 全角英数字は使用しないでください.
- ASCII 記号の全角版 (例: `＋`, `＿`, `！`) は使用しないでください.

正しい記述例:
- Ctrl + Alt

誤った記述例:
- `Ｃｔｒｌ＋Ａｌｔ`

## 見出し, 構成

### 見出しの書き方

- 見出しは体言止めで統一してください.
- 見出しの末尾に句点は付けません.
- 見出しのレベル (階層) は, 内容の包含関係に基づいて適切に設定してください.

正しい記述例:
- `## インストール手順`
- `### 環境変数の設定`

誤った記述例:
- `## インストール手順を説明します.` (体言止めでない, 句点がある)
- `## インストール手順` の直下に `#### 環境変数の設定` (見出しレベルを飛ばしている)

### 図表, コードブロックの参照方法

- 図, 表, コードブロックには通し番号とキャプションを付けてください.
- 本文中で図表を参照する際は, "図 1", "表 2" のように番号で参照してください.
- 図表は, 本文中で参照する位置のできるだけ近くに配置してください.

正しい記述例:
- 図 1 にシステム構成を示します.
- 表 2 の結果から, 処理速度が向上していることがわかります.

誤った記述例:
- 下の図にシステム構成を示します. (番号で参照していない)
- 以下のテーブルの結果から, 処理速度が向上しています. (番号で参照していない)

## 例文

以下に, 本文書で定めたルールに従った例文を示します.

```text
Python 3.12 のインストール手順を以下に示します. まず, パッケージ一覧を更新し, 追加で必要なライブラリー (build-essential など) を導入します. その後, Python の公式サイトよりソースコード (.tar.xz 形式) をダウンロードし, 解凍してからインストール作業を進めます. 詳細は README.md を参照してください.
```

```text
C++ と Rust は, 共にコンピューターのハードウェア性能を極限まで引き出せるシステムプログラミング言語ですが, その設計思想とアプローチには明確な違いがあります.

C++ は 1980 年代からの長い歴史を持ち, OS (operating system), ゲームエンジン, 金融取引システムなど, 膨大な既存資産とエコシステムを有しているのが最大の強みです. Object-oriented programming や template metaprogramming による柔軟で自由度の高い記述が可能ですが, メモリ管理の多くが開発者に委ねられています. そのため, 不適切なポインター操作による buffer overflow や memory leak といった, セキュリティリスクの高いバグが発生しやすいという課題があります.

一方, Rust は安全性と並行性を優先して設計された現代的な言語です. "Ownership" と "borrowing" という独自のシステムを導入することで, ガベージコレクションに頼ることなく, コンパイル時に厳密なメモリ安全性を保証します. これにより, CPU 使用率が 100% に近い高負荷な状況でも, 実行時のオーバーヘッドなしに C++ と同等のパフォーマンスを実現しつつ, 多くのメモリ関連バグを未然に防ぐことができます. また, 標準のパッケージ管理ツール Cargo の利便性も高く評価されています.
```
